# redis的原子性

## 事务

1. multi 开启事务
2. exec 执行事务
3. discard 放弃事务
4. watch（监控，乐观锁）

- multi开启事务之后 就可以写每一行命令。

- 每一行命令都是先去排队，只有exec后才开始轮流执行。

- 如果执行discard 则丢弃写的指令。
- watch在multi之前使用，监视一个或多个key，如果在此期间key被改动，之后的exec直接失败返回null，保证并发安全！

是半原子性：

1. 入队失败：写命令相当于入队。如果命令写错，则当场报错，执行exec时，所有命令都做废。
2. 入队成功，执行报错：假如有a,b,c命令成功入队，exec执行了，a成功执行,b逻辑错误，不执行，c成功执行。redis只会执行a，c。b不执行，不支持事务回滚！！ 如果是mysql事务，则会回滚a c的成功。

所以redis的原生事务，只保证隔离型，在执行这批命令的时候，别的客户端无法插队，但不保证原子性。



## 原子性的实现

1. 压缩成单命令
   - redis的核心命令执行是单线程的
   - 把复杂操作压缩成一个原生命令 在单线程里执行无法被打断 天然有原子性
2. 乐观锁watch  cas思想
   - 开启watch盯着key 是除了lua外
3. lua脚本
   - 保证了多个步骤的原子性 执行期间别的客户端进步不来 比如redisson的分布式锁底层
