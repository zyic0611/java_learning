# 数据类型与表范式

## 数据类型选择

核心原则是 保证数据精度和未来扩展性的前提下 选择占用空间最小的数据类型 

更小的数据类型 则一页B+树节点 16KB 可以存下更多的数据 减少磁盘IO。

### 整型

TINYINT 1字节  SMALLINT 2字节  MEDIUMINT 3字节  INT 4字节  BIGINT 8字节

1. 不要无脑用int
   - 状态枚举 布尔值的情况下 使用TINYINT 在海量数据的情况下可以缩小表体积和索引体积。

2. INT（11）
   - 括号不是代表可以存储的数字 而是显示宽度 只有配合zerofill使用才有意义
   - int永远只占4字节 最大范围21亿多 

3. 主键选择
   - 通常使用  无符号的 BIGINT UNSIGNED  。尤其是使用雪花算法生成分布式ID。

### 浮点数与定点数

1. 浮点数： 
   - float 4字节 double 8字节 底层使用二进制近似值存储 存在精度丢失问题 金融慎用。

2. 定点数：
   - decimal（M，D） mysql底层是把decimal转化为字符串存储 所以是精确的 M代表总位数 D代表小数位数

### 时间类型

1. DATETIME ：

   - 8字节  mysql5.6以后优化到5字节

   - 存什么就是什么 与时间无关 范围广

2. TIMESTAMP:

   - 存储1970年1.1 0时 UTC 以来的秒数 

   - 写入的时候会把当前时区时间转换为 UTC秒数

   - 读取的时候在根据当前数据库的时区配置转换为本地时间

   - 问题： 只有4字节 只能存到2038年1月19日

架构选择： 不涉及多时区 优先使用datatime 。 多时区 可能会直接使用bigint存储unix时间戳 在应用层里进行时间转换 避免数据库里时区计算的开销。

### 字符串类型

1. char(N)  固定长度

   - 不管有几个支付 都会用空格补全到N个字符的长度

   - 优势： 没有存储碎片 处理速度快

   - 适合： UUID 手机号 身份证号

2. varchar(N) 可变长度

   - 按照字符实际长度存储。 除了数据本身 还需要1-2字节记字符串真实长度。

   - 不能N写的很大 虽然在disk上按照实际长度存储。 但是如果查询涉及到orderby groupby 需要在内存中创建临时表 内存临时表会按照N的最大可能长度分配内存 。过大的N会导致内存消耗 降级为磁盘临时表 引发性能抖动

## 表设计 范式与反范式

### 3nf

1nf： 字段不可再分

2nf：消除部分依赖  是解决主键内部的问题 

主键的意义是一个主键确定一条数据 当需要多个键确定一条数据的时候 就需要使用复合主键。

比如主键由ab两个属性构成 还有两个非主键cd。

知道ab 才知道c 则c完全依赖于复合主键。

知道a就知道d 则d部分依赖于复合主键。

消除的办法就是拆表。部分依赖的那一部分拆成子表。

违反2nf的问题： 

- 严重的空间冗余：本来一个表一行存的数据 现在需要重复存储很多次。
- 更新异常：如果数据更新 则相关数据全部更新 如果漏了几行 就会出现数据不一致



3nf：消除传递依赖  解决非主键的问题

一个c键 依赖于 b键 b键又依赖于主键a 则c传递依赖于a。

### 反范式

一般不遵循3nf 如果严格遵循 会导致一个selct 要jon多表 效率低。

一般以空间换时间 适度增加冗余字段 查数据只需要单表查询 甚至走覆盖索引就行。

但是数据一旦修改 就需要更新多表。解决办法一般采用消息队列orbinlog 异步解耦 既保证了主业务不阻塞 也保证了系统的最终一致性。
