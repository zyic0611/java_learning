# MySQL整体架构

架构的核心特点是 职责分离。 

采用了插件式的存储引擎  把查询数据和数据存储分开。

宏观上分为三层结构：

连接层 server层 存储引擎层。



## 连接层

java后端应用连接mysql 就是通过连接层打交道。



连接处理：

采用半同步/异步处理方式。 为每个建立的连接分配一个thread。

为了避免频繁创建和销毁线程带来的开销。 内部维护一个线程池。



授权认证：

验证账号密码 以及验证用户操作的权限。



## Server层

核心功能的实现： 包括内置函数（时间 数学 日期） 跨存储引擎功能： 视图 触发器 存储过程。

按照一条 `SELECT` 语句的执行顺序，Server 层包含以下几个关键组件：

1 查询缓存： mysql8.0废弃

- 机制：查询请求 先去缓存里找 
- 废弃原因：失效机制很严苛 只要表里有一条数据更新 所有关于这个表的缓存全部清空     对于频繁读写的业务 缓存命中率低 增加了系统维护缓存的开销    对于高并发架构   很差

2 解析器

- 如果没有命中缓存 进入解析器
- 识别出SQL语句中的关键字  SELECT FROM &表名 列名
- 判断SQL语句是否符合语法  关键字拼写正确、少逗号
- 最后产出一个抽象语法树 AST

3 预处理器

- 解析器只判断语法对不对  也就是关键字 不管语义。
- 预处理器管表、列是否存在 别名是否有歧义 等语义的问题

4 处理器 server层灵魂

- 决定sql语句如何执行最快  mysql使用的是基于成本的优化器 CBO Cost-Based Optimizer
- 决定定是否走索引 以及使用哪个
  - 如果是多表join 决定表的连接顺序 驱动表和被驱动表的选择
  - 重写sql语句 消除无用条件 常量的提前计算
- 产出一个执行计划 explain关键字看到的就是优化器后的结果

5 执行器

- 拿着优化后的执行计划 执行
- 执行前 会再次判断该用户是否有操作该表的权限 （上次判断在连接层）
- 执行器本身不负责读写顺序 它通过 存储引擎提供的api获取和修改数据



## 存储引擎层

负责数据的真正存储和提取

插件式架构： server层通过统一的api和存储引起ing交互 可以为不同的表指定不同的存储引擎

InnoDB ：现在的默认引擎。支持事务、行级锁、外键。

MyISAM：不支持事务，不支持行级锁（只有表锁），但在全表扫描和某些读密集场景下速度较快。现在很少作为核心业务引擎使用了。



## 一条select语句的完整生命周期

```
SELECT * FROM user_info WHERE age = 25 AND status = 1;
```

1. **连接器：** 验证身份，建立连接。
2. **解析器：** 提取出 `SELECT`、`user_info`、`age`、`status`，检查语法，生成语法树。
3. **预处理器：** 检查 `user_info` 表存不存在，`age` 和 `status` 这两个字段存不存在。
4. **优化器：** 假设表里有 `age` 的索引和 `status` 的索引。优化器会计算成本，发现走 `age` 索引扫描的行数更少，于是决定使用 `age` 索引，生成执行计划。
5. **执行器：** 调用 InnoDB 引擎接口，说：“给我查 `age=25` 的第一条数据。”
6. **存储引擎：** InnoDB 拿着条件去 B+ 树索引里搜索，找到数据后返回给执行器。
7. **执行器：** 拿到数据后，在 Server 层判断 `status = 1` 是否成立（如果是索引下推机制，这一步会有所不同）。成立则放入结果集，然后继续向存储引擎索要下一条，直到遍历完毕。最终将结果返回给客户端。
