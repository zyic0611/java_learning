# 02_innoDB内存架构

innoDB架构设计的核心思想是 磁盘I/O太慢 能用内存解决的问题尽量在内存里解决 最后再通过顺序I/O异步刷盘。



## Buffer Pool

内存中最大的区域 作用主要是 缓存磁盘上的数据页和索引页。   

查询数据的时候 会先去bufferpool里面看有没有  有的话直接返回 走内存读。 没有的话 去disk读取 并且加载到 buffer pool里。

修改数据的时候 也是先修改buffer pool里的数据 此时变成脏页 再由后台线程定期刷入磁盘。



- 数据结构： 按Page 默认16KB 为单位进行管理

### 内存管理算法



 使用改进后的LRU 最近最少使用

传统LRU 问题： 如果发生全盘扫描 大量冷数据会瞬间塞满 把热点数据挤出去 导致缓存污染

冷热分离方案： 把LRU按比例分为两部分 : New Sublist 热数据区 默认5/8   Old Sublist 冷数据区 3/8

运行机制：

新读取的页 先放在old区的头部 在old区停留超过一定阈值 且再次被访问 就会移动到new区的头部 



## change buffer

写缓冲区。 实际上优化的是一段时间内的多次update。 也就是一段时间内对同一个索引页的多次修改。一次修改是一次随机I/O 多次就是多次随机IO。 所以适合写多读少。

把多次随机IO 立即执行 变成 延迟执行 批量合并。

核心目的是优化  **非唯一 普通索引**的写入性能。

### 产生原因

 在更新数据的时候 由于簇状索引（主键） 的叶子 放的是所有数据 所以簇状索引一定会被更新。

以及普通(二级)索引会被更新。 普通索引的非叶子结点就是普通数据 叶子节点是主键ID。

由于普通索引的数据 这些页大概率不再buffer pool  并且在磁盘中 也在不同的页里 要更新 就必须从磁盘里随机读取这些页 产生 随机IO（如果是在内存里 连续数据就可以使用cache机制 预先读出连续的数据 io效率就比较高）



### 工作机制

需要更新一条数据 并且数据不再buffer pool中 innodb不会直接去disk读取这个页面 而是把更新操作记录在change pool中。

merge：

当这个非唯一索引页 因为其他查询操作 被读入 buffer pool or 后台线程定期执行 时候  innodb会把change buffer里关于这个非唯一索引的记录 合并到真正的页面上 把多次随机I/O 合并为一次操作



### 限制场景

只适用于 非唯一索引 。 因为唯一索引 在写入的时候 必须把数据页读入内存 来校验是否冲突。 页都读入内存 直接修改buffer pool就行了 change buffer失去了意义。





## Adaptive hash index AHI

自适应哈希索引   

B+树再快 查询的时候 也会经过几层节点的寻路 查询效率是logn。 

- 工作机制：
  - InnoDB 会持续监控表上各索引页的查询频率。如果它发现某些确定的值被极度频繁地访问（比如相同的等值查询），它就会在内存中基于 B+ 树的这一部分**自动构建**一个哈希索引。

- 特性：

  - 它是“自适应”的，由数据库内部隐式创建，外部无法手动干预（只能选择开启或关闭）。

  - 它**只支持等值查询**（如 `=`, `IN`），不支持范围查询（`>`, `<`）。



## log buffer

日志缓冲区 崩溃恢复的缓冲区。

为redo log 重做日志服务。 为了保证事务的持久性 又不能让每次修改都同步刷盘 开销巨大。 

innodb采用WAL write-ahead logging 日志先行技术

### WAL

修改数据 先在buffer pool里修改 产生 脏页

然后把修改的物理日志 redolog 写入内存的log buffer

事务提交的时候 再把log buffer里的日志 顺序io的方式 快速刷入disk里的redo log。

只要日志落盘 就算db宕机 重启后也能恢复。

### 刷盘策略

**（必考参数 `innodb_flush_log_at_trx_commit`）**：

- `0`：事务提交时，只把日志留在 Log Buffer 里，每秒钟后台线程统一刷到 OS Cache 并落盘。（性能最高，但如果 MySQL 宕机，最多丢失 1 秒的数据）。
- `1`（默认值）：每次事务提交，都必须把 Log Buffer 里的日志写入 OS Cache 并**立刻调用 fsync() 落盘**。（最安全，严格保证 ACID，但高并发下磁盘 I/O 压力大）。
- `2`：每次事务提交，把日志写入操作系统的 OS Cache（Page Cache），由操作系统决定何时落盘，同时 MySQL 后台线程每秒也调用一次 fsync()。（折中方案，MySQL 挂了数据不丢，只有服务器所在操作系统宕机或断电才会丢失约 1 秒的数据）。



## 总结：

执行一条update 先看buffer pool里有没有该数据页 

没有并且是非唯一索引 就记入change pool

有的话 就在内存里修改 同时生成redolog放入log buffer

如果该数据频繁的被等值查询 就使用ahi进行加速。
