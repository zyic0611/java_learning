## MHA集群架构

如果mysql主库 master主班直接烧了 物理宕机 彻底连接不上 如果没做高可用 系统会直接瘫痪。



MHA：Master High Availability 主节点高可用。

如果主库主板烧毁 依靠人工介入： DBA需要切主库 改代码配置... MHA就是把这个过程自动化。

**故障转移工作流**：

### 发现与隔离 防止脑裂

- 精准探测： 

  - MHA的manager节点定时ping主库 如果主库没响应 不会立刻行动 因为可能只是网络抖动 。

  - 它会让别的从库也去ping一下 大家都认为主库挂了 才真挂了。

- 隔离与VIP漂移：
  - 确认主库挂了后 MHA会立刻将绑定在主库上的VIP（virtual ip 虚拟ip）摘除。
  -  因为后端代码配置连接数据库是这个ip(重点 如果配置的是vip 就不需要真的去改后端配置)。
  - 摘除vip是为了防止主库复活 导致后端应用同时往两个主库写数据(脑裂)



### 极限抢救数据

MHA最核心的地方 尽最大的努力保证不丢失数据 。

假设主库是mysql 进程崩溃。但是物理机的系统还存活者（还可以ssh连接上）

主库挂的瞬间 可能还有部分binlog 归档日志没有发给从库。

此时mha可以通过ssh登录到快死的主库 像翻遗物一样 找到没发送的binlog文件拷贝出来



### 选主和补齐

- 比对日志：
  - MHA去看剩下的所有从库 谁的中继日志最新 也就是谁的数据最接近老主库 就把谁选定位新主库。
- 数据回放:
  - mha会把抢救出来的binlog 在新主库上回放重演 这一步做完 就可以保证新主库的数据和老主库死前一致 数据没丢失

### 重构集群

将从库正式升级为master 解除只读权限 并且把vip漂移绑定到新master身上。 这样后端不需要重启 依靠连接池的重启机制即可。

mha还会自动修改其他从库的同步配置 让他们知道谁是新的主库。



### 问题

mha架构虽然很经典 抢救机制很完善。 但是它强以来ssh互相面密登录。 在云原生和docker容器化环境下很难搞。 

并且它是perl脚本写的 开源社区几乎不维护了。



## mycat

mycat是基于proxy的中间件。伪装成mysql让java去连接 。作为一个路由分配的节点。

大厂爱问： 既然所有流量都打向mycat mycat自己宕机了怎么办 那不是成为系统最薄弱的单点。

业界经典 高可用架构：**Keepalived + HAProxy + MyCat + MySQL。**

### 架构流转过程

- 最底层： mysql主从集群
  - 负责真正的数据存储 结合mha保证数据库本身的单点故障恢复。

- 中间层：多台mycat节点
  - 至少部署2台mycat 配置了相同的分库分表固规则 底层连接同一套mysql

- 流量分发层：HAProxy
  - java应用不能写死连接哪一个固定的mycat 否则一台挂了就得修改代码。 
  - 在前面加一个HAProxy（专业的软负载均衡中间件） HAProxy负责把java后端应用发来的sql请求 轮询打到mycat a/b上 实现mycat层的负载均衡。

- 终极高可用层: Keepalived

  - 如果haproxy挂了咋办 所以haproxy也需要部署两台(主备模式) 这两台机器上会运行keepalived服务 

  - keepalived会对外虚拟出一个vip。 java项目 数据库连接写的实际上是vip。
  - 主haproxy宕机 keepalived就会瞬间把vip漂移到备用haproxy上。

  

- 总结： 

  为了一个分库分表 引入4层结构 运维成本高 排查链路长 所以mycat慢慢被边缘化。



## ShardingSphere 

Apache的顶级项目 是一个庞大的Database plus生态。

### 双核引擎

使用 JDBC和 Proxy的完美配合

- **ShardingSphere-JDBC（客户端轻量级）**：纯 Java 框架，无中心化，直连数据库。适合高并发的微服务核心链路，性能损耗极低。

- **ShardingSphere-Proxy（服务端代理层）**：像 MyCat 一样独立部署，支持 MySQL 和 PostgreSQL 协议。适合非 Java 语言（如 Python、Go），或者给 DBA 用可视化工具（如 Navicat）去直接连接管理整个分片集群。

大厂混合架构落地：在真实生产中，往往是核心 Java 微服务使用 JDBC 模式（追求极限性能），而外围系统或数据分析团队使用 Proxy 模式（追求跨语言和管理便利），两者共用一套底层数据库。



### 分布式治理

客户端jar包（jdbc） 最怕修改配置 数据库只要增加了一个节点 所有几十个微服务都需要重启。

所以shardingsphere引入了**治理中心** (配合ZooKeeper和Nacos)

- 动态治理：只需要在nacas或者shardingsphere的控制台修改一下路由规则或者加一个数据源 配置会立刻推送到所有运行着 shardingsphere-jdbc的jvm进程，无需重启 直接生效
- 熔断与限流： 集成了类似于sentinel的功能 底层发现某个从库压力太大 可以直接在中间件层面把发往那个库的sql 降级或者限流。
- 分布式事务无缝融合： 分库分表必遇分布式事务 shardingsphere内部集成了seate(阿里开源的事务中间件) 只需要简单配置 就可以自动接管跨库事务的XA or AT模式
