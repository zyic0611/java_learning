# Question

Q：在高并发场景下，多个事务同时读写同一行数据。MySQL 默认的隔离级别是“可重复读（Repeatable Read, RR）”。 *提问：* **InnoDB 是如何通过 MVCC（多版本并发控制）底层机制来实现“可重复读”的？**（提示：需要你讲清楚 `Undo Log` 的版本链，以及 `Read View` 是在什么时候生成的，怎么做可见性判断的）。

A：

mysql在修改数据时 不会直接删除旧数据 而死在undolog里记录版本链。从最新的数据指向旧数据，构成一个版本链条。当事务开启后，用户进行第一次select 此时会拍一次快照 生成read view 以read view中的信息为标准，

“Read View 生成时，会维护几个关键变量：当前活跃（未提交）的事务 ID 数组 `m_ids`，最小活跃事务 ID `min_limit_id`，以及即将分配的下一个事务 ID `max_limit_id`。 判断规则非常严密：

1. 如果数据版本的 `trx_id < min_limit_id`，说明事务早就提交了，**绝对可见**。
2. 如果 `trx_id >= max_limit_id`，说明是生成 Read View 之后才开启的新事务修改的，**绝对不可见**。
3. 如果介于两者之间，就判断 `trx_id` 是否在 `m_ids` 数组里：如果在，说明修改这条数据的事务还没提交，不可见（顺着 Undo Log 往下找）；如果不在，说明已经提交了，可见。”

Q：很多人说 InnoDB 的 RR 级别彻底解决了“幻读（Phantom Read）”问题，**这个说法完全正确吗？** 如果不完全对，在什么极端场景下依然会发生幻读现象？

A：并不完全对。首先解决了幻读是在两个情况下解决。第一就是快照读的情况下，通过undolog和readview配合，在第一次select开启时拍摄快照，保证读取数据一致。避免了幻读。 第二就是当前读的情况下，通过临健锁 将数据和数据间的间隙上锁 防止插入或者修改数据 避免了幻读。但是在读写交叉的时候 解决不了幻读。先读数据 此时拍摄快照，其他事务在之后插入一条数据 并提交了事务。然后本事务开启当前读，准备修改数据，由于当前读是去数据库里找最新的数据，不是根据快照，本事务成功更新了其他事务插入的数据。此时还没幻读。本事务更新了这行数据 则这行数据的事务id就变成本事务 然后本事务再执行查询 发现看到了这个数据 因为自己修改的数据总是对自己可见 此时就幻读了。

Q：假设有一张 `user` 表，`id` 是主键，`age` 是一个**普通的二级索引（非唯一）**。表里现在有三行数据，`age` 分别是 10, 20, 30。 现在开启了一个事务 A，执行了这样一条语句：`UPDATE user SET name = '张三' WHERE age = 20;` *提问：* 在 RR 隔离级别下，**这条 SQL 语句在底层到底加了哪些锁？**（提示：结合 Record Lock 行锁、Gap Lock 间隙锁 和 Next-Key Lock 临键锁来回答）。如果此时事务 B 想插入一条 `age = 15` 或者 `age = 25` 的新数据，会被阻塞吗？为什么？【等值查询的的上锁逻辑】

A：

**Next-Key Lock（临键锁）：** 首先锁定 20 所在的区间，也就是前一个值和它本身的左开右闭区间 `(10, 20]`。此时如果插入 `age=15` 会被这个锁阻塞。

**Gap Lock（间隙锁）：** 因为 `age` 是非唯一索引，MySQL 不知道 20 后面是不是还有别的 20，所以它会继续向后扫描，直到遇到第一个不为 20 的值（也就是 30）。然后对这中间的间隙加锁，即 `(20, 30)` 的纯间隙锁（不包含 30）。此时如果插入 `age=25`，会被这个间隙锁阻塞。

**Record Lock（主键行锁）【最容易漏的加分项】：** 别忘了，修改了数据，InnoDB 最终是要去聚簇索引（主键索引）上执行更新的。所以，它不仅锁住了二级索引上的 `age=20`，**还会顺藤摸瓜找到 `age=20` 对应的那行数据的主键 ID，给那个主键索引也加上行锁（Record Lock）。**”

Q：如果是范围查询>=20 上锁逻辑是什么呢

A：

**第一步：定位起点并加锁** InnoDB 首先通过 B+ 树快速定位到第一条满足 `age >= 20` 的记录，也就是 `age = 20`。

- 它会加上一个 **Next-Key Lock（临键锁）**，锁住它的前一个值到它本身的左开右闭区间，即 **`(10, 20]`**。
- *同时，去主键索引上，把 `age=20` 对应的那行真正的数据加上 Record Lock（行锁）。*

**第二步：向右顺藤摸瓜（因为是 >=）** 因为条件是 `>= 20`，InnoDB 必须继续向右扫描。下一个遇到的是 `age = 30`。

- 满足条件，继续加 **Next-Key Lock**，锁住区间 **`(20, 30]`**。
- *同样，去主键索引上，把 `age=30` 对应的记录加上行锁。*

**第三步：寻找到无穷大（防幻读的终极手段）** 遇到 30 就结束了吗？绝对没有！由于是范围查询，为了防止你在后面“偷偷”插入 `age = 40` 或者 `age = 100` 的幻影数据，InnoDB 会继续向右扫描。

- 此时它扫到了 B+ 树数据页中的一个特殊虚拟标识——`Supremum pseudo-record`（代表无穷大）。
- 于是，它毫不客气地加上了最后一把 **Next-Key Lock**，锁住 **`(30, +∞)`**。



Q：假设你执行了一条极其普通的语句：`UPDATE user SET age = age + 1 WHERE id = 1;` *提问：* 在这条语句的执行过程中，**`Undo Log`、`Redo Log` 和 `Binlog` 分别扮演了什么角色？** *大厂致命追问：* 为了保证 InnoDB 引擎层的 `Redo Log` 和 Server 层的 `Binlog` 数据一致，MySQL 引入了**“两阶段提交（Two-Phase Commit）”**。请你推演一下这个两阶段提交的流程。如果在第一阶段（Prepare）成功后，还没来得及写 Binlog 机器就断电了，MySQL 重启后会怎么处理这笔事务？

A：

Undolog是修改数据的时候建立版本链  里面记录了反向操作 age+1 就记录-1 万一事务执行到一半就报错了 用来回滚 保证原子性A 

RedoLog 就是记录 物理上 哪页的内容被修改成了什么 追加写的形式记录【顺序IO 

BinLog 就是记录这句sql语句 或者具体行的数据变更 主要用于主从复制和数据恢复 

追问A：两阶段提交 就是先在redolog上记录  然后打上prepare标志 然后再binlog里记录 然后将prepare标志修改为commit。 mysql查看redolog时 发现标志是prepare 再查看binlog里 发现没有与redolog对应的日志 就不会重写redolog里prepare的内容。 不会恢复事务。

Q：

我们知道磁盘 I/O 是非常慢的，如果每次 UPDATE 都去直接改磁盘里的数据页，数据库早就卡死了。 *提问：* InnoDB 是如何通过 **Buffer Pool（缓冲池）** 和 **WAL（Write-Ahead Logging，预写式日志）** 技术，把随机写变成顺序写，从而极大地压榨出极高并发写入性能的？如果修改的数据还在 Buffer Pool 里（脏页），还没刷入磁盘时机器宕机了，数据会丢吗？为什么？

A：

每次修改数据的时候 把磁盘里的数据页复制到内存的缓冲池里  在缓冲池里修改 等晚上没什么使用的时候 再统一刷入磁盘。

WAL机制 在事务提交时候 indb只会把修改动作记录到redo log里 并且把redo log顺序写入磁盘 顺序写磁盘的速度堪比写内存

只要redolog成功罗盘 事务就算成功  内存里的脏页  后台线程会以一定的频率 满满的 异步刷到磁盘的数据文件中

宕机的时候不会丢失数据 可以用redolog把没来的及刷盘的脏页推演恢复出来【Crash Safe机制

Q：线上环境通常是一主多从架构。 *提问：* 请简述 MySQL 主从复制的底层基本原理（提示：涉及 Binlog、IO 线程、Relay Log、SQL 线程）。 *大厂极限实战：* 默认的异步复制有一个致命缺陷：如果主库刚把事务提交，还没来得及同步给从库，主库的物理机就彻底炸了（磁盘损毁）。这时候主从切换会导致数据丢失。**在大厂的金融级支付业务中，我们通常会开启什么级别的复：

A：

主从复制依赖3个核心线程

1. 主库的dump线程 当从库连接的时候 主库还开启这个线程 负责把binlog传给从库
2. 从库的i/o线程 接受主库传来的binlog 写入从库本地的relay log【中继日志中
3. 从库的sql线程 不断读取relay log中的日志 在从库中重放执行 完成数据同步

默认的异步复制 在主库磁盘损坏的时候 确实会丢失数据。所以金融级业务中 开启 半同步复制。

主库执行完事务 写完binlog 不会立刻告诉客户端 执行成功。必须等待 直到至少有一个从库的i/o线程接受到了这段binlog 并给主库返回一个ack确认包 主库才会完成提交 并且响应客户端。牺牲了威晓得性能，但是保证了就算主库爆炸 数据也能绝对安全到达了至少一个从库 实现了强一致性的兜底。
