重做日志 主要是为了保证 Crash-Safe能力 也就是崩溃恢复 主要是确保事务的持久性。  

一般是 预防断电情况 只要事务提交了 就算数据库断电了 数据也不会丢。



## 为什么出现

修改的数据为什么不直接写到磁盘里 还要先去写日志。

核心矛盾 **磁盘的随机IO太慢了** ：

- buffer pool ：mysql为了提高性能 所有的数据增删改查都是在内存进行的。

- 脏页 :内存的数据修改后 和disk里的数据不一致。 叫做脏页 如果每次修改后就把脏页刷回disk 由于真实data可能分布在disk的各个角落 属于随机IO
- WAL:预写式日志 :innoDB 先在内存中修改数据 然后把修改了什么记录到redolog里 。 写redolog属于顺序I/O  末尾追加 速度快
- 异步脏刷：redolog只要写入成功 mysql就会告诉客户端提交成功。 至于脏页 后台可以打开专门的异步线程 慢慢刷入磁盘。

**所以本质上就是由轻量级的顺序写 替换了笨重的随机写。**

## 物理日志

redolog记录格式非常底层  类似于将 表空间id为10 数据页号为20 偏移量为100的位置更新为123

- 为什么设计成物理日志：

因为物理日志恢复速度快 比sql语句快（解析语法树 检查权限 ) 只需要覆盖对应物理页数据就行， 所以宕机恢复很快的原因。



## 数据结构

空间是固定大小 类比圆环，所以是环形数组和双指针。

- 写指针：记录写到哪里
- 检查点 记录有多少数据成功刷盘 刷盘成功 该指针就会移动 腾出redolog空间

因为是环形数组 所以采用循环写 



## Os Cache

OS会拿一部分内存 当作磁盘缓存 这就是OS Cache。

任何程序 不管是mysql java python 都没有权限直接操控磁盘硬件。 所以写文件操作 默认都是写到os cache内存里 就立马返回成功。

- write() 只是写入os cache

- fsync() 强制操作系统 把oscache里的数据 落盘到disk

推出： 写binlog redolog 也是写入os cache 无法直接操作disk





## 刷盘策略

0 :事务提交 不做任何操作。 后台线程每1s 写日志写进oscache 并且调用fsync()刷入disk

1:事务提交马上 写入oscache 并且fsync()

2: 每次提交事务 马上写入oscache 但是fsync()的时间由os决定


## 必要性
redolog 搭配binlog
如果只有binlog 没有redolog

- binlog的缺点 不知道结果
     - 断电重启后 内存里有很多脏页 去询问binlog binlog 只知道逻辑上修改了什么 
     - 但是不知道脏页写进了硬盘里没有 
     - binlog无法记录物理页的状态 innodb没办法进行精确的崩溃恢复
     - 也就是分不清断电重启后 哪些脏页需要重新刷进去
   - redolog
     - 是循环写的 有个核心概念叫做 checkpoint 
     - checkpoint之前的数据 代表刷入了 这部分的redolog空间就可以被覆盖 
     - 断电重启后 innodb会寻找最新的checkpoint 然后顺序往后执行
