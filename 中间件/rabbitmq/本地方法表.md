# 本地方法表

问题：在医疗项目中，你用 RabbitMQ 异步投递订单来实现削峰 。如果 Java 生产者刚刚把消息发出去，网络就断了，你怎么确保这条订单消息真的安全到达了 RabbitMQ 的 Broker 里？

消息可靠性投递里的发布确认机制 只能知道 生产者消息是否成功发送出去给了队列。

但是如果java服务器宕机 重启后 内存里的数据完全丢失了 也不知道发了什么消息出去 就算收到了ack回复也没用 完全失忆 所以需要一个备份工具。

必须在java生产者端做好兜底。采用**本地消息表**

1. 在创建业务订单的同时 在同一个数据库事务里 往一张 本地信息表里 插入一条 消息记录   状态设置为发送中
2. 由于在同一个数据库事务中 保证发送操作和记录操作的强一致性 也就是订单和消息表数据的强一致性
3. 然后java端在调用rabbitmq api 发送消息给消费者
4. 在java代码里 开启rabbitmq的发布确认机制 只有rabbitmq返回了ack回答 才代表消息发送成功 把消息表中的记录标记为发送成功
5. 兜底防护：如果网络断了 java 没有收到ack  此时开启一个定时任务 每隔1min去扫描一下本地消息表 把那些 状态为发送中的消息 重新投递 保证了网络无法怎么抖动 信息都不会丢失



## 总结：

发布确认机制 提供了网络传输成功与否的依据 。

mysql的本地消息表利用落盘和定时任务  提供了不会丢失的重试数据源  

二者结合 保证了百分百 发送端的可靠性保障。



## 深入分析

发布确认机制 只是一个通知器 只是一个依据 没有任何记忆和存储的能力。 没有解决网络问题的能力。

### 问题：

1. 如果生产者刚发送消息 断网 mq没收到消息。 此时java服务器健康 没有宕机。

2. 没收到 则没有ack确认回复

3. java生产者超市等待后 抛出异常

4. 抛出异常的方法执行结束 存在于ram里的消息对象失去了引用 会被gc清楚

5. 就算 网络恢复 java想重发 也没有消息的数据去重发。



### trycatch:

用catch接收异常 使用while一直重发 直到ack：

如果断网期间 10000个请求 进来，则有1w个请求在while循环里 tomcat线程池瞬间被打满 后续所有网页都打不开 导致OOM java服务器宕机。



### 本地消息表的价值：

所以内存非常不可靠 不管是服务器宕机 网络断开 抛出异常 线程结束 内存回收 最终的结果都是消息在内存中消失。

所以本地消息表并不仅仅是防止宕机 是所有异常:断网 超时 宕机 报错 的唯一物理硬盘备份。



### 架构设计：

1. 本地消息表为唯一依据
2. 发送消息给mq
3. 断网 异常 不在内存重试 直接结束线程 释放cpu资源
4. 硬盘上有 发送中的记录
5. 网络恢复后 定时任务会把这条数据拿出来 重新发送
