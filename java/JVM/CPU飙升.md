

## cpu 飙升100% 怎么救火

排查4步骤

1. top命令 发现PID 123【进程号 为xxx的CPU的占用率很高
2. top -Hp 123  -H看线程 -p指定进程号 发现这个java进程 有一个线程id为 456的吃光了CPU
3. printf "%x\n" 456  把线程号转换为16进制 因为jm打出来的底层日子 线程号全部都是16进制 为101e
4. jstack 123 | grep -A 20 101e 导出123进程的堆栈 搜索101e这个线程  打印他的前后20行日志
5. 结果会出现此线程正在执行的java代码位置 打开对应位置排查

“通过 `jstack` 拿到代码行号后，根据我的经验，通常分为两种截然不同的情况：

1. **业务线程飙升：** 说明代码里大概率出现了**死循环**（比如刚才聊到的看门狗没配好导致死循环），或者是在处理极其复杂的正则表达式（正则回溯陷阱），这就需要去 review 具体的业务代码。
2. **GC 线程飙升（极其高频）：** 如果我发现占用 CPU 最高的居然是 JVM 的 `VM Thread`（垃圾回收线程），那说明系统其实不是在疯狂计算，而是**内存快爆了（发生内存泄漏或突发大对象）**。JVM 正在拼命地进行 Full GC，导致 CPU 被拉满。这时候光看 CPU 就不够了，我会立马切去用 `jmap -dump` 导出内存快照，用 MAT 工具去查到底是什么对象把内存撑爆了。”

“当然，如果公司允许引入第三方诊断工具，在真实的线上救火场景中，我首选阿里开源的 **Arthas**。遇到 CPU 飙升，直接连上 Arthas，敲一行命令 **`thread -n 3`**，它就能一键直接把当前 CPU 占用率最高的前 3 个线程的堆栈信息打印出来，连转 16 进制和 `grep` 都省了，极大地提高了线上排障的效率。”







运行 `java -jar arthas-boot.jar` 后，控制台会列出你电脑上当前所有正在运行的 Java 进程。

1. 你会看到列表中有一个叫 `CpuSpikeDemo` 的进程，前面有个数字序号（比如 `[1]`）。
2. 你输入这个序号 `1`，按回车。此时 Arthas 就成功挂载到这个出问题的 JVM 上了！屏幕上会出现彩色的 `arthas@PID>` 提示符。
3. **敲下那句终极咒语：** `thread -n 3` 按回车！

你会瞬间看到，Arthas 直接把刚才那个占了 99.9% CPU 的 `main` 线程揪了出来，并且**极其精准地把具体报错的代码行号（比如 `CpuSpikeDemo.java:6`）高亮打印在了你的屏幕上！**





“如果系统没 OOM，CPU 也没飙升，但是有些接口就是卡死一直不返回，可能是什么情况？”

- **你的秒答话术：** “这种极其安静的卡死，通常是发生了**线程死锁**。排查极其简单，不用去猜，直接用 `jstack` 打印堆栈，拉到文件最后，如果发生了死锁，JVM 会极其明确地打印出一句 `Found one Java-level deadlock` 并把互相死锁的线程列出来。如果在 Arthas 里，直接敲一行 `thread -b`（b 代表 block），它就能直接找出那个占着锁不释放的罪魁祸首。”
