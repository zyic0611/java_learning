# List接口

一般有三个实现类, LinkedList ArrayList Vector。

List是有序的集合，意思是存储和添加顺序有序。并且允许元素重复和有多个null值。

## ArrayList

1. 懒加载:
   - 在被new ArrayList<>()时，底层没有分配长度为10的数组，而是给了一个默认长度为0的空数组，第一次调用add的时候，才会去初始化一个长度为10的数组。
   - 为了节省内存，创建了不用浪费内存。

2. 1.5倍扩容机制

   - 底层触发扩容机制grow() 扩容1.5倍

   - 为什么是1.5倍 是通过位运算>>1 原来长度+原来长度的一半 位运算比除法快得多

   - 扩容不是在原来基础上加长 是开辟一个全新的数组 然后哟哦那个底层的System.arraycopy() 复制过去 代价高昂

3. 选择性

   - 在内存中连续，非常符合Cpu的缓存预读机制，实际开发中首选arraylist。
   - LinkedList节点分散，会导致严重的cpu缓存未命中。
   - 理论上arraylist中间插入很慢，实际开发中链表由于要先遍历定位，可能遍历难度到，速度不一定更快。

4. 初始化:

   - 如果能预估数据量，初始化的时候要指定初始容量，频繁的触发扩容机制代价很高。

5. 查询：

   - 支持随机访问 时间复杂度为O(1) 链表不支持 为on
   - 尾部插入很快，中间插入删除很慢 为o(n) 
   - 链表在已知节点插入删除很快，只需要断开引用，o1

   

## LinkedList

1. 折半查找优化：
   - 查找指定位置的元素 get(index) 位o(n)
   - 实际上不是每次都要从头到尾遍历，源码里做了优化，每次会判断index在前半段还是后半段 前半段从头往后着 后半段从尾往前找。最坏情况也只是o(n/2)

## CopyOnWriteArrayList

cow，AraayList是线程不安全的，在读多写少的环境下，可以使用这个半线程安全的cow。

- 读时不加锁，读操作完全并发，性能很高。

- 写操作，称为写时复制。先复制一份，给线程写修改后，在覆盖原来的。 通过加锁来保证同一时刻只有一个线程在复制和修改。

- 适合黑名单，配置信息，路由表这种半年不改一次，读很高频繁。

缺点：

- 内存占用，写的时候内存里会同时存在两个数组。
- 数据一致性，写的时候别人读的还是旧数据，只能保证最终一致性，不能保证实时一致性。 也就是可能读到旧数据，弱一致性。

## 一边遍历一边删除元素

1. for循环： 可以 容易出bug 漏删/错位
2. foreach :不可以
   - 底层使用Iterator 迭代器内部维护一个计数器 expectedModCount (预期修改的个数 )
   - ArrayList 维护了一个 modCount 实际修改次数
   - 用在增强for里使用list.remove 删除一个元素，modCount++ 但expectModCount没变，
   - 迭代器在下一次取元素的时候会检查两个数相等吗 不相等旧报错 报并发修改异常
3. 正确方法 Iterator.remove() 回去修改expectModCount 或者 倒序遍历 从后往前遍历+删除 不会影响前面没有遍历的元素的位置

## 删除指定下标的元素

1. ArrayList : 
   - 调用System.arraycopy() 把数组往前挪 时间慢 除非删除最后一个
   - list.remove(index)
2. LinkedList
   - list.remove(index)
   - 删除o1 但是找到元素on 总的还是on
3. 最好用的方法 把最后一个元素复制到要删除的位置 然后删除最后一个元素  在不考虑顺序的情况 是最快的

## ArrayList的线程不安全

1. 同时执行list.add

   - 源码：elementData[size++]=e
   - 单线程 把值赋给size位置 size++
   - 多线程： a和b线程同时读到size=5 都把自己的值先后读入5位置 然后各自执行size++ 
   - 此时size=5位置的值是b的 但是size=7 size=6的地方是空值
   - 导致了出现空值和数据丢失

2. 数组越界 没有扩容

   - 同时添加两个元素 1 2 同时进行扩容检查 发现不用扩容
   - 扩容检查:ensureCapacityInternal(size + 1); 在add操作前先执行
   - a线程进来先检查size+1 够 
   - 还没写 换b b也扩容检查 也够
   - 此时再轮到a来写 
   - b再写 旧数组越界了。

3. 多线程同时触发扩容

   - ArrayList扩容设计新数组创建 旧数组拷贝
   - 多线程同时扩容 内存中可能多处好几个扩容后的副本而 `ArrayList` 的内部 `elementData` 引用在这些副本间跳来跳去，最终导致有的数据在老数组，有的在新数组，整个列表状态彻底错乱。

4. 解决方法：

   - **方案 A**：用 `Collections.synchronizedList(new ArrayList<>())`，给所有方法强行加锁。

   - **方案 B**：用 **`CopyOnWriteArrayList`**。利用“写时复制”思想，读写分离，适合读多写少。

   - **方案 C**：直接在业务层代码加锁（最土但最稳）。
