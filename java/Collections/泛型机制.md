# 泛型机制

是编译时期的安全类型检查机制

1. 泛型擦除
   - 在编译过程中 会将所有泛型信息查处 并在相应位置插入强制类型转换代码

2. 泛型通配符
   - <? enxtends T> 上界通配符  允许T及其之类。适合读的时候，只能读T 
   - <? super T> 下界通配符 可以存放T及其父类 适合写 只能写T

3. 自动装箱与拆箱
   - 装箱 Integer.valueof()
   - 拆箱 intValue()

## 为什么不能用基本数据类型 

1. 泛型擦除后变成Object 基本数据类型不继承Object 
2. 并且java框架设计初衷是操作对象



## 性能问题

1. 循环中会大量自动装箱 拆箱 产生大量无用对象 增加GC压力

###  自动装箱/拆箱的性能陷阱

- **基本类型 `int`**：就是裸露的黄豆，直接抓一把塞进碗里，速度极快，不占额外空间。

- **包装类 `Integer`**：给每颗黄豆都精美包装在一个盒子里。 如果你写了一个循环，在里面不断地把 `int` 变成 `Integer`，就相当于你每秒钟都在制造几万个小盒子。这些盒子用完即丢，**保洁阿姨（GC）**就得疯狂跟在你屁股后面捡盒子。



``` Long sum = 0L; // 注意：这里用的是包装类 Long for (int i = 0; i < Integer.MAX_VALUE; i++) {    sum += i; // 致命陷阱！ } ```

sum+=1 实际上被编译为

sum=Long.valueOf(sum.longValue()+i)

每次循环都会 经历一次 拆箱 获取值 装箱 创建新的Long对象存long 会产生21亿Long对象 。

**最佳实践**：在局部变量、循环计算、方法内部，**优先使用基本类型**（`int`, `long` 等）。只有在必须使用集合类（如 `List<Integer>`）或表示数据库“空值（null）”时才使用包装类。

### IntegerCache 缓存

在 Java 程序中，小整数（比如 -1、0、1、10）的使用频率极高。为了节省内存和提高效率，Java 预先做了一批“马甲”（对象）放在池子里，谁要用直接给同一个。池子就是`Integer` 内部静态常量池。

**源码范围**：默认是 **-128 到 127**。

**源码触发点**：只有通过 `Integer.valueOf(int)`（即**自动装箱**）时才会走缓存。如果是 `new Integer(10)`，则会绕过缓存强行创建新对象。

这里就走了自动装箱 把int 127 int 128 装成Integer     

而Integer是包装类 比较两个包装类 如果不用equals 用== 一般都是不相等的

但是127 这一块 是使用了缓存 指向同一片内存。

**扩展知识**：

- `Long` 也有类似的缓存池。
- `Character` 缓存了 0 到 127。
- **`Float` 和 `Double` 没有缓存池**（因为小数是无穷的，没法缓存）。
- 缓存上限 127 可以通过 JVM 参数 `-XX:AutoBoxCacheMax=<size>` 修改。
