# 自动装配

## 启动类的复合注解：@SpringBootApplication

- @SpringBootConfiguration 底层就是@Configuration 把当前启动累标注为一个配置类 让Spring容器知道里面可能定义了Bean

- @ComponentScan  从启动累开始扫描所有 用@Component @Service @Controller标签的类 都放进Spring容器 
  - 如果业务代码没写在启动类的同级或者子包下 是扫描不到的
- @EnableAutoConfiguration 核心 实现自动装配

##  @EnableAutoConfiguration

去扫描所有引入的jar包中的目录 寻找自动装配类得清单。

### 清单

pom.xml 是Maven用的清单 在这上面写 需要的依赖 

Maven会去仓库把对应的.jar包下载到我的项目里

application.yml 是SpringBoot用的清单 告诉SpringBoot 引入的依赖要怎么用 

.jar包中会有清单 清单里写了很多全限定类名 不管用不用 都在这

### @Conditionnal 条件规则

清单里有几百个配置 不可能全部加载 此时就使用Conditional 注解来做判断 按需加在

- @ConditionalOnClass 判断代码里有没有相关类 有的话才会去实例化相关配置
- @ConditionalOnMissingBean 判断容器里有没有某个bean 如果用户自己没有配置 那么就加在一个默认的。 **保证了用户的自定义配置优先级高于自动配置**
- @ConditionalOnproperty 判断配置文件【application.yml 里有没有写相关的属性



## Starter 机制

自定义Starter工程。

1. 建工程 命名规范
   - 新建一个Maven/Gradle 工程
   - 官方命名：spring-boot-starter-xxx
   - 第三方： xxx-spring-boot-starter
2. 引入依赖
   - 在pom.xml中引入springboot的基础依赖 和封装的中间件的核心依赖
3. 编写自动配置类
   - 写一个类 写上@Configuration注解
   - 通过@Bean 实例化自己的核心组件
   - 大量使用@Conditional注解
     - @ConditionalOnMissingBean 防止用户自己定义配置组件被覆盖
     - @EnableConfigurationProperties 把业务系统的application.yml里的配置读取到一个Properties类中
4. 注册自动配置类
   - 在resource/META-INF/spring/ 目录下写一个`org.springframework.boot.autoconfigure.AutoConfiguration.imports` 的文件。
   - 把自动配置类的全限定名写入 com.zhuyicheng.xxx.MyxxxAutoConfiguration
   - 最后用mvn clean install 打包
   - 业务系统只要在pom引入starter enableautoconfiguration就会自动读取improts

## SpringApplication.run() 的流程

1. 准备环境 获取OS的环境变量 JVM参数 读取application.yml文件的配置信息
2. 创建上下文 【AppicationContext
   - 根据引入的依赖 决定创建什么类型的容器
   - 比如spring-boot-starter-web 创建servlet web类型容器
3. 准备上下文 
   - 把第一步的环境绑定到Context里 把启动类加载到容器里 
4. 刷新Context
   - 调用Spring框架的refresh方法
   - 自动装配开始根据Conditional注解按需加载Bean并且注入依赖  IOC&DI
   - if is Web SpringBoot启动内置的Tomcat
