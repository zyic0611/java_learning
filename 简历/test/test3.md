## 一、 JUC 与 JVM 进阶排错 (检验底层基本功)

1. **[JUC 内存泄漏]** 在医疗平台中，为了方便在各个业务层获取当前登录医生的信息，很多项目会使用 `ThreadLocal` 来存储用户上下文。请问 `ThreadLocal` 为什么会引发内存泄漏？在搭配线程池使用时，你会在代码里做怎样的规范化处理来避免这个问题？

ThreadLocal 的原理是每一个线程都有自己的ThreadLocalMap key为创建的ThreadLocal对象 value为医生的信息 key为弱引用 value为强引用 如果这个线程是普通线程 执行完任务就被销毁 那么 key和value都会被释放 不会发生内存泄漏。

但是日常开发中使用的都是线程池，线程使用完不会被销毁，会被回收，则key没被对象指向 是弱引用 会自动释放 此时 唯一指向value的key被释放了 由于value是强引用 无法自动释放  强引用被线程池的长生命周期拉住 此时就发生了内存泄漏。

为了避免这个问题 生产环境中 有铁律 必须在try finally代码块呢 显式调用remove方法 。

保证http请求处理完毕 tomcat线程归还给线程池之前 当前线程挂在的用户上下文都被清空

2. **[Redis 分布式锁续期漏洞]** 昨天我们聊了 Redisson 看门狗机制防死锁。但是，如果你的 Java 业务节点发生了 **“假死”**（比如发生了极长时间的 Full GC 停顿，或者 JVM 被操作系统挂起），看门狗线程也跟着一起被冻结了，无法续期导致锁提前释放，被其他线程抢走。等假死恢复后，原来的线程继续执行，就会出现两个线程同时持有一把锁的严重 Bug。对于这种极端情况，业务层面有办法兜底吗？

​	这是一个经典的分布式脑裂问题。在JVM假死 导致看门狗失效的极端情况 redis层面的校验是不够的，假死恢复的线程并不知道锁已经易主 会继续执行并发写操作。

要解决这个兜底这个漏洞 最终的防线必须下沉到数据库层面。结合我在我在项目中的实践 引入乐观锁CAS机制或者版本号校验。

比如扣件库存的时候 update stack set count=count-1 ,version=version+1 where id=? and version= 原版本号

即使发生了锁脑裂 性线程拿到锁执行成功 修改version 假死苏醒的旧线程继续去执行sql 会因为version不匹配导致更新失败。

利用数据库底层的原子性 保证了数据的一致性。

## 三、 RabbitMQ 与 MySQL 核心机制 (考验数据一致性)



1. **[MySQL 深度分页]** 医疗平台有海量的历史问诊记录。如果医生需要在后台系统翻看非常靠后的记录，比如执行了这样一条深度分页 SQL：`SELECT * FROM records LIMIT 1000000, 10`。MySQL 会非常慢，因为它要扫描前 100 万条数据然后丢弃。如果你是底层研发，你会如何利用**“延迟关联 (内连接)”** 或 **“子查询”** 的技巧来优化这条深度分页 SQL？

limit 1000000,10 的意思就是跳过前一百万条 然后拿出来10条数据。 mysql底层就是全盘扫描一百万条+10条，效率很低。

最好的办法是先通过子查询 在id索引树上查出这10条数据 在id索引树上查找效率高 再通过连接 根据id 与原表做内连接 然后拿出这10条完整数据。 因为走了覆盖索引

select * from records t1

inner join (select id from recodes  limit 1000000 10) as t2 

on  t2.id=t1.id

总结就是 子查询ID 延迟关联
