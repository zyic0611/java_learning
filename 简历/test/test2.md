## 一、 JUC 并发编程底层 (检验内功)

1. 

   **[AQS 原理]** 你的简历里写了深入理解并发编程，熟练运用 AQS 及锁机制 。我们平时用的 `ReentrantLock` 就是基于 AQS 实现的。请你用大白话描述一下，当三个线程同时去抢一把 `ReentrantLock` 锁时，AQS 底层的 `state` 变量和那个双向队列（CLH）分别是怎么工作的？

   AQS的核心是一个volatile变量state和一个虚拟的双向队列CLH。

   首先三个线程会尝试着用cas操作去修改state变量 肯定只有一个修改成功了 则state变量此时就是1 。

   修改成功的线程 就把它设置为锁的主人。

   当其他两个线程发现state变量变成1了 则不会一直自旋消耗CPU。

   会被包装为Node结点 加入到CLH的尾部，然后调用park()方法让这两个线程挂起阻塞等待。

   当一个线程执行完毕 把state状态变为0 。才会调用unpark 唤醒队列头节点，让它出队 重新尝试着用cas获取锁。

   

2. **[线程池极限场景]** 刚才我们聊过把 AI 异步任务的线程池拒绝策略设置为 `CallerRunsPolicy`。假设你的 Tomcat 主线程（大堂经理）真的去执行这个极其耗时的网络 I/O 任务了，在这个任务完成之前，如果有新的前端 HTTP 请求打到这个 Tomcat 线程上，会发生什么？这会带来什么隐患？

主线程去执行这个极其耗时的网络I。O任务 相当于tomcat主线程被长期阻塞。

这个时候如果突发流量进来 导致大量任务被拒绝 tomcat的线程池会被瞬间打满 【tomcat自带的用来接收http请求的线程池被迅速耗尽

tomcat线程池被打满 会让服务器陷入假死状态 无法接受任何外部http请求。

此时是一种极其极端的被动降级和自我保护 。

在生产环境中 更倾向于在网关层就把多余流量拦截 而不是等tomcat线程卡死。

## 二、 Spring 框架与 MySQL 调优 (实战踩坑)

   **[Spring 事务机制]** 简历提到你解决过 `@Transactional` 事务失效等复杂场景 。除了内部方法调用会导致失效，如果在你的业务代码里，你在一个带 `@Transactional` 的方法内部 `try-catch` 捕获了一个 `SQLException` 并且**没有向上抛出**，这个事务最后还会回滚吗？为什么？

   @Transacational是基于AOP代理实现的 代理逻辑在catch到异常的时候 才会执行回滚。

   如果在业务代码里把异常捕获了 没有thorw抛出 aop代理层不知道发生了什么 他会认为方法正常执行 直接提交事务。

   导致了严重的脏数据。 

   如果业务确实需要捕获异常日志 又需要事务回滚 

   则需要在catch块里手动调用强制标记回滚。

   

2. 

   **[MySQL 联合索引]** 你熟练掌握 MySQL InnoDB 索引结构与 EXPLAIN 。假设在医疗系统中，我们为了查询患者历史就诊记录，建了一个联合索引 `(patient_id, doctor_id, visit_date)`。如果我的 SQL 是 `SELECT * FROM records WHERE patient_id = 1 AND visit_date = '2026-03-26' ORDER BY doctor_id`。这个查询能完全用上联合索引吗？会触发 `Using filesort` 吗？

   只能部分走联合索引，先通过patientid走二级索引树 然后找出id为1的一系列结点 在通过索引下推 isp 通过visit_date = '2026-03-26' 过滤一些数据 然后在通过剩下的结点 回表查询。

   根据最左前缀原则，最左边是patientid 并且使用了patientid的等值查询 则查询出的结点是天然按照doctorid排序的 不需要内存排序。 不会触发Using filesort

   

## 三、 医疗项目深度拷问 (中间件与高可用)

1. 

   **[RabbitMQ 消息丢失防范]** 在医疗项目中，你用 RabbitMQ 异步投递订单来实现削峰 。如果 Java 生产者刚刚把消息发出去，网络就断了，你怎么确保这条订单消息真的安全到达了 RabbitMQ 的 Broker 里？（提示：生产者确认机制/本地消息表）

   这是一种极端的网络断裂情况 导致刚发出去的消息丢失。只考mq自身的持久化是不够的。

   必须在java生产者端做好兜底。采用**本地方法表**

   1. 在创建业务订单的同时 在同一个数据库事务里 往一张 本地信息表里 插入一条 消息记录   状态设置为发送中
   2. 由于在同一个数据库事务中 保证发送操作和记录操作的强一致性 也就是订单和消息表数据的强一致性
   3. 然后java端在调用rabbitmq api 发送消息给消费者
   4. 在java代码里 开启rabbitmq的发布确认机制 只有rabbitmq返回了ack回答 才代表消息发送成功 把消息表中的记录标记为发送成功
   5. 兜底防护：如果网络断了 java 没有收到ack  此时开启一个定时任务 每隔1min去扫描一下本地消息表 把那些 状态为发送中的消息 重新投递 保证了网络无法怎么抖动 信息都不会丢失

2. 

   **[缓存双删的极端漏洞]** 针对预约取消场景，你做了 500ms 异步延迟双删 。我们之前聊的是第二次删除失败怎么兜底。但如果**第一次**删除 Redis 缓存的操作就因为网络超时失败了，你还会继续去更新 MySQL 吗？如果不更新，业务就卡住了；如果更新，缓存里就会一直残留脏数据。你会怎么处理这个两难境地？

   这是针对可用性【用户使用体验 和 一致性 发生冲突的业务场景。

   如果一发生异常 就抛出 不让更新mysql 那么redis只要网络抖动 用户则一直无法操作。C端业务来说 不允许。

   1. 如果第一次删除redis就因为网络抖动失败了 不能阻塞主业务 继续更新mysql
   2. 因为在 高可用架构中 数据库永远是第一位 业务流程必须往前走
   3. 但是为了解决缓存里的脏数据的问题 采用 **异步重试兜底**
   4. 当网络抖动时 异步发送 要删除的key传给rabbitmq 让一个专门负责删除redis里key的消费者 在后台 不断尝试删除 
   5. 牺牲了短暂的一致性 用户可能会读取到旧的数据 但是保证了核心业务链路的高可用 保证了用户的核心操作不被卡死

   

3. 

   **[Redisson 分布式锁底层]** 你引入 Redisson 保障核心支付接口幂等性 。假设一个用户发起了支付请求，业务代码获取到了 Redisson 锁，但在执行业务逻辑的过程中，这个方法又调用了另一个同样需要获取这把锁的内部方法（锁的可重入场景）。Redisson 底层是怎么判断“这是同一个线程再次要锁”并允许放行的？它是依靠什么数据结构记录重入次数的？

   Ridisson实现可重入锁 主要靠 redis的hash数据结构和 lua脚本。

   redisson底层不采用String结构  采用hash结构。

   redis key为 锁名字

   hash key 客户端UUID+线程ID redisson可以根据这个确定是那个线程+的锁

   hash value 重入次数

   1. 一个线程第一次来+锁的时候 redisson执行lua脚本 在hash结构中记录UUID：ThreadId为key value初始化为1。

   2. 当需要重入的时候  【业务方法内部调用了其他需要同一把锁的方法 redison会再次执行lua脚本，发现锁被占用 但是占用着的uuid 线程id 和申请线程一致 

   3. 此时不会阻塞 value+1 
   4. 同理 unlock 释放锁的时候 底层也是调用lua脚本 value-1
   5. 直到value为0 代表这个分布式锁被释放 则调用del命令从redis中删除锁

   

   

   

## 四、 架构工程化 (场景设计)

1. 

   **[MQ 架构选型反问]** 针对 GPU 推理耗时的痛点，你设计了 MinIO + RabbitMQ 异步流转架构 。有个很现实的工程问题：既然 RabbitMQ 可以发消息，为什么你不直接把医疗影像转换成 Byte 数组（比如 10MB 的图片），直接塞在 RabbitMQ 的消息体里发给 GPU 节点？为什么非要多引入一个 MinIO 组件来做存储？

   1. rabbitmq的物理瓶颈 ：

      - 设计初衷是处理轻量级的控制流信号 kb级别。
      - 把图片数据直接塞进消息体 会瞬间打满rabbitmq broker的网络带宽
      - 还会导致大量的内存占用 和极长的GC停顿
      - 会阻塞队列里其他轻量级业务消息
      - 甚至引发集群OOM崩溃

   2. 业界标准的 凭证模式

      - 大文件异步处理 业界标准架构方案是凭证模式

      1. 图片放进minio
      2. 寄存处给小凭证
      3. 只需要把小凭证封装成消息 投递给mq
      4. 底层的ai算法节点拿到凭证消息后 就拿着url去minio拉去突破进行推理

      实现了数九路 和 控制流 的物理隔离

   3. 业务上的持久化闭环

      -医学影像本身就有很强的持久化和归档需求 这些图片不仅ai要看 医生复诊也要看 存在minio 一举两得
