   redis节点突然宕机的一致性问题： 

   1. 脚本执行层：首先lua脚本本身保证了预扣减的原子性，意味着里面的逻辑要么全部执行 要么不执行，如果执行到一半突然宕机，本次操作在持久化文件里是不生效的时候，重启后数据仍然一致，相当于请求失败，让用户前端重试即可。
   2. 节点故障层：真正的风险在于lua脚本执行成功了 扣件了redis里的库存。结果redis主节点还没来得及同步给从节点 从节点就挂了。当redis发生宕机的时候 此时发生主从切换 新的主节点丢失了刚才的lua脚本里的记录。 如果业务需要高可用性，可以把aof持久化策略配置为always或者everysec 来尽量减少数据丢失。但是还是可能导致少量扣减redis里库存数据丢失。
   3. 数据库兜底：但是并没有把redis当成最终存储，而是当成前置防御，真正的底线在mysql。讲只要lua脚本成功，java就会发送消息给消费者让其扣减库存。 即使redis挂了 导致缓存里的库存>实际库存，但是真正扣件库存操作打向mysql的时候，mysql采用乐观锁校验，保证不会超卖。 同时，可以利用定时任务，或者坚挺mysql的binlog 将数据库的真实的号源反向同步，重新预热到redis里，定时强制覆盖redis里的脏数据。

3. 

   **[缓存一致性]** 针对预约取消场景，你基于线程池实现了 500ms 异步延迟双删以保障 MySQL 与 Redis 最终一致性 。为什么是 500ms？如果第二次删除 Redis 缓存的操作由于网络波动失败了，你会怎么兜底？

   因为是延迟双删，第一步删除redis，第二步修改mysql信息，第三步休眠500ms，因为在第二步期间可能还有线程去读redis信息 读不到 然后去读mysql的旧数据 写会redis 第四步再次删除redis数据 避免脏数据。休息500ms就是保证第二步其他线程的读mysql，写会redis操作完成。在第四步可以彻底删除旧数据，避免数据不一致。

   500ms不是绝对的，实际项目中，需要评估项目中**读数据业务逻辑的最大耗时。**通过日志统计出读接口的P99耗时【%99的请求都在这个时间完成 然后在这个基础上+1,200ms作为延迟时间，能保证大多数读旧写新的操作在这个窗口期里完成，又不会让线程池里的任务堆积太久。

   **[Elasticsearch 机制]** 你搭建了基于 Elasticsearch 的检索引擎来支撑医疗资源的筛选 。ES 底层的倒排索引是什么结构？当用户输入一段复杂的病情描述去搜索专家时，ES 是如何进行分词并打分排序的？

   

   倒排索引结构： 从逻辑上来说 倒排索引是哈希表结构的。key为词 value为documentId的列表，但是如果new一个哈希表对象放进内存中，很容易OOM。 所以在disk里采用有序字符数组term dictionary来存储词典，按字典序有序排序，然后数据落在disk里的倒排表posting list里，并且用咆哮位图压缩过。里面是documentId。然后由于在disk里二分查找 数据量大 磁盘I/O次数多，在内存里会有term index,采用fst数据结构，是极致压缩的前缀树，存储的字典的目录，然后找到disk里这个词前缀对应的偏移量，再进行二分查找一小段数据即可。

   ES分词并且打分：复杂的病情，采用的是match，会使用分词器依照词典把病情描述进行切分，将得到的词去倒排索引里寻找对应的document。

   打分主要考虑3个纬度

   1. TF 词频 当前词在document里出现的频率 越高打分越高
   2. 逆文档频率 IDF 这个词在index里越稀有 得分权重越大
   3. 字段长度归一化：document越短 得分权重越大 因为短文本命中代表相关性强。
