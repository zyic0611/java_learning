# http

http的特点就是无状态  不记得是谁的信息。

所以发明了 cookie 和session 每次发送http请求 都会在请求头【header里加入cookie 相当于是身份证

http1.0:

问题：每次发送一个http请求 都必须tcp三次握手 请求完毕都得四次挥手断开连接

如果网页里有100张图片 就必须经历100次tcp握手 网络开销大



http1.1:

引入了长连接 只要建立了一次tcp三次连接 就可以在这个连接上发送多个http请求  节省了握手时间

问题： 对头堵塞 服务器按照请求顺序返回响应 如果第一个请求很慢 后面的小请求也必须阻塞等待 整个通道卡死



http2.0:

1. 引入了多路复用 在一个tcp连接上 并发多个http请求 和响应
2. 二进制分帧 把数据打碎成了很多小的二进制帧 并且打上Stream ID 这样所有数据在TCP包混着发送 浏览器收到后 再根据编号拼接 消灭了http的队头阻塞
3. 头部压缩  之前每次请求头部都带几百字节重复的cookie和header 2.0时代双方各自维护一本字典 重复的header直接发一个索引号 比如1就是cookie:xxx 节约带宽



http3.0:

抛弃了TCP

TCP的问题：绝对保证顺序 底层网络丢了一个包 TCP协议栈就会强制把后面的数据扣下 等待丢失的包重传 意思就是 TCP层丢了一个包 http2.0所有并发的请求都卡死

http3.0采用基于UDP的QUCic协议 不管丢包和顺序 在用户态制剂实现了可靠性 让并发请求互不干扰
