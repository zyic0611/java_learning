# 滑动窗口与拥塞控制

## 滑动窗口

本质上是接收方告诉发送方 自己当前的缓冲区的可用大小。

发送方会根据这个数值 动态的调整自己的发送速度。

避免发送的数据溢出 导致丢包问题。

## 拥塞控制

逻辑是投石问路 试探性发车。

1. 慢启动 指数级试探  
   - 一开始不知道网络有多顺畅 先发一个数据包试探 
   - 如果顺利收到ack 就double
   - 目的就是用最短时间摸清网络的运载极限
2. 拥塞避免 线性刹车
   - 不可能无限的double TCP内部设置一个阈值
   - 到达这个阈值后 就不会double 每次只+1
   - 变成线性增长小心试探 直到网络发送拥堵/丢包
3. 快速重传 三次警告
   - 发现丢包 旧做法 等待超时 浪费时间
   - 新做法： 只要接收方发现中间少一个包
   - 就连续发送3个同样的重复ACK
   - 发送方只要收到三个重复ACK 就判定数据丢失 立刻重传 不等待超时定时器
4. 快速恢复
   - 虽然收到了3个ACK 说明丢包 但是网络没有彻底瘫痪 
   - 此时TCP会把拥塞窗口 /2 把阈值也调整到/2后
   - 直接进入拥塞避免的线性增长
   - 保证了网络在轻微波动下的极高吞吐量



## 真实发送窗口大小

实际发送窗口大小=min(接收方允许的窗口大小，网络允许的拥塞窗口)
