# Redis delete&update

解决**缓存与数据库一致性**

## Redis 修改和删除的区别

### 并发带来的顺序错乱：

假设有两次修改请求（请求 A 和 请求 B）几乎同时发生：

1. **请求 A** 修改数据库为 `值 A`。
2. **请求 B** 修改数据库为 `值 B`。
3. 由于网络抖动，**请求 B** 先到达了 Redis，把缓存改成了 `值 B`。
4. 随后 **请求 A** 到达了 Redis，把缓存改成了 `值 A`。

简单理解为修改数据库的顺序是A、B 修改Redis的顺序不一定是A、B。

此时就会出现redis和db数据不一致的问题。

而如果是删除redis缓存，查redis为空，则去查db 此时一定是最正确的。

### 计算消耗

修改缓存，如果需要更新的是一个复杂的对象，需要先查出来旧对象，修改后，再序列化写入redis，在写多读少的情况下非常浪费性能。

如果是删除缓存，动作极快，并且只有有人要读数据时，才回去db里加载（lazy load.

### 取消预约的逻辑

如果状态，库存 都在redis里有。

- 对于状态，不要set 直接删除，让下一次查询去数据库里查最新的。

- 对于库存，先执行db里的操作，再删除redis里的缓存。并且异步出发一个逻辑，重新计算库存并且同步到redis里（或者等下一次下单发现redis里没缓存自动从db加载 lazyload
