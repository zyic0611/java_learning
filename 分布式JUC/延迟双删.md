# 延迟双删

## 问题

在所谓的 **Cache-Aside（旁路缓存）** 模式中，我们通常的操作是“先删缓存，再改数据库”。但在高并发下，会出现两个线程读写之间的问题.

1. **线程 A（写操作）**：删除了 Redis 中的缓存 `Stock=100`。
2. **线程 B（读操作）**：发现缓存没了（Cache Miss），去读数据库，读到了**旧值** `100`。
3. **线程 A（写操作）**：更新数据库，把 `Stock` 改成了 `99`。
4. **线程 B（读操作）**：把刚才读到的**旧值** `100` 写回了 Redis。

**结果**：数据库是 `99`，缓存是 `100`。除非缓存过期，否则用户看到的永远是错的。这就是**脏数据入库**。

## 操作

1. 先删除缓存，保证所有查询都会去数据库里查最新数据。
2. 修改数据库。
3. 休眠，为什么要休眠一会 因为 A线程改库需要时间，此时也会有别的B线程来查缓存，查不到缓存自然会去查库，库里仍然是旧数据，此时就会把旧数据写入缓存。
4. 再删除一次缓存，保证缓存里不会有脏数据。



## 线程池异步

#### **逻辑步骤**

1. **主线程**：删除 Redis 缓存。
2. **主线程**：更新数据库。
3. **主线程**：将“延迟删除任务”提交给 `ScheduledExecutorService`。
4. **主线程**：`return true`（用户感觉不到延迟）。
5. **后台线程**：睡 500ms -> 执行第二次删除。

## MQ

#### **逻辑步骤**

1. **写操作**：删除 Redis 缓存 + 改数据库。
2. **写操作**：发一条消息给 RabbitMQ。
3. **RabbitMQ**：通过“延迟插件”或者“死信队列”，让消息在 500ms 后才投递。
4. **消费端**：接收到消息，执行 `redis.delete(key)`。
