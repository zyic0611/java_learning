# 向量数据库和检索进阶

## 与es mysql对比

传统数据库： 靠的是精确匹配（mysql） or倒排索引（es 使用TF-IDF或者BM25） 他理解不了语义

向量数据库：靠的是语义相似度 计算余弦相似度或者欧氏距离 文本变成空间中的坐标点 

**常见选型：** Milvus（大厂标配，支持十亿级向量）、Qdrant/Weaviate（轻量级，云原生友好）

## 缺点

向量相似度虽然懂语义 但是对专有名词 特定数字 代码片段不友好。

比如提问一个 ：报错代码 NullPointerException_0x88 怎么解决？

向量模型完全没见过0x88 就会把他变成一个完全偏离的向量坐标 根据这个向量坐标 召回的都是一些不相干的 完全错过了正确答案。 这时 传统的ES关键词 倒排索引检索 效果却很好。



## 多路召回！

1. 先使用向量检索 利用词嵌入模型 负责抓去语义相近的文档 topk个 泛化能力强
2. 关键词检索 比如使用BM25算法 利用es的传统分词 负责抓去 字面精准匹配的文档 特别是特定ID 特定人名的内容
3. 融合重排（RRF算法） 两路分别找出TOP10 然后使用RRF算法重新打分排序。合并。如果一个文档在两边排名都很高，就喂给llm.

## 提问

Q：

现在我们的客服业务上线了 RAG 系统，但发现了一个头疼的问题：用户的提问往往非常口语化，甚至缺斤少两。比如我们知识库里存的文档标题叫**『2026年春节退换货物流延迟补偿标准』**。

但用户往往只会输入两个字：**『退货』** 或者 **『赔钱吗？』**。 如果只用这两个字去搜向量库或者 ES，召回的效果都极差，因为信息量太少了。

**请问，在不修改知识库本身的前提下，你会如何在用户的查询阶段（Query 阶段）做文章，来优化这种‘极简提问’或‘口语化提问’的检索效果？有没有听说过相关的工程策略？**”



本质上也就是 强化问题。

如果用户的提问太短 太口语化 导致信息量很少。 需要用大模型把用户的问题 翻译 and 扩写一些。 这个叫 Query Rewrite 查询重写技术。

1. HyDE llm的盲答扩写

不着急查向量数据库 因为文本过短。 先用llm写一个假设性答案，比如：

- **系统后台偷偷发给 LLM 的 Prompt：** `用户提出了一个词“退货”，请你假设你是一个电商客服，写一段关于退货政策的介绍。`

- **LLM 生成的假答案：** `一般情况下，平台支持7天无理由退货。如果商品存在质量问题，退货运费由商家承担。生鲜类商品不支持退换...`

再把fake ans 转成向量 查向量数据库 由于fakeans语义丰富 在多维空间里 就可以精准匹配了。

2. 同义词生成器 Multi-Query

用llm把用户精简的提问扩充

- **做法：** 让大模型帮用户把一句话，扩展成 3 句意思相近，但侧重点不同的话。
  - **系统后台 Prompt：** `请把“赔钱吗”这句话，改写成 3 个不同视角的电商售后问题。`
  - **LLM 输出：**
    1. `延迟发货的经济补偿标准是什么？`
    2. `申请退款和赔付的流程怎么走？`
    3. `商品损坏会有额外的现金赔偿吗？`

现在有了3个标准的query  同时去库里搜【并发检索 然后再利用prf算法 合并去重。

3. 意图与实体提取 

结合元数据挂。 大模型先做一个文本信息提取 提取用户的意图json文件。然后根据这个json 生成一个精准的向量数据库查询 也就是带标量的查询（结合元数据挂载） 排除其他干扰，很金主。



所以标准回答：

“对于口语化和极简提问，我会引入 **Query Rewrite (查询重写)** 机制。具体来说，我会使用大模型作为检索前置的‘翻译官’。比如采用 **HyDE 策略**，让模型先生成一段包含丰富行业词汇的假设性答案，用这个长文本的向量去检索；或者使用 **Multi-Query** 扩写出多个相似问题进行并发召回。同时，如果知识库已经做了良好的结构化（Metadata），我会用大模型先提取用户的**业务意图实体**，将其转化为标量过滤条件。通过大模型本身的能力来增强检索的命中率。”
