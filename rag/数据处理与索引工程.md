# 数据处理与索引工程

## 为什么做chunking

1. 由于llm上下文长度有限 并且输入越长 费用越高 推理越慢 所以一定要给文本做chunking【文档切割。

2. 如果用超长文本做embbeding 和用户一个短文本向量 做相似度计算 语义会被稀释 找不准 所以必须切块

## 粒度

太小： 包含的信息很少 llm拿到后缺少上下文

太大：包含了太多无用信息 增加token消耗 干扰大模型注意力

一般在256-1024Toekn之间 大概几百个汉子 结合换行符来切

## 规避语义切割的技术

### 主动规避策略

1. 滑动窗口/重叠切分overlap: 最基础也是必须的。 比如 Chunk A 是 0-500 字，Chunk B 不是从 501 开始，而是从 400 开始（400-900）。中间 100 字的 Overlap 保证了交界处的语义连贯。
2. 按规则or标点符号：优先按\n if段落还是太长 按照。 尽量保证一句话or一段话是完整的
3. 大文档摘要/父子文档重构： 先切大块【父文档 再切小块【子文档 存向量的时候 存子文档 因为搜的准。但是喂给llm 父文档 因为要保证上下文完整。



## Embedding技术 选型

是深度学习模型 比如bert衍生模型。 作用是把文档映射喂一个多维的高维浮点数空间。 

特点是语义相近的句子 在多维空间里 距离近。

选型：

中文常用 BAAI（智源）的BGE系列 or ali的Qwen-Embeding。

指标：

- 向量维度越高 比如1536 语义表达越丰富 但是存储成文本 检索越慢 反之(384)
- 上下文长度 ：这个模型支持最多能把多长的句子转成向量 一般是512 or 8192token
- 业务适配度 通用模型在垂直领域效果不一定好 可能需要用内部业务数据对embedding模型进行微调。



## 提问：

Q：在我们处理那本 500 页《后端架构手册》时，遇到了一个 Bug。

> 手册第 30 页有一个大标题叫 **【订单服务的降级策略】**，下面跟着一段具体的配置代码和重试机制的描述。
>
> 业务同学提问：‘订单服务怎么配置重试机制？’ 结果系统没找对答案。我们排查后发现，切割的时候，**【订单服务的降级策略】** 这个标题被分到了 Chunk 1，而 **具体的配置代码和重试机制** 被分到了 Chunk 2。因为用户的提问里有‘订单服务’，所以系统去查的时候，Chunk 2 里只有干巴巴的代码，没有‘订单服务’这四个字，导致相似度极低，没有被召回。
>
> **请问，如果遇到这种‘标题和正文被跨 Chunk 切断，导致正文丢失了上下文（不知道这属于哪个业务线）’的情况，你有什么在工程上容易落地的解决方案吗？**”

首先overlap是必须的 但是overlap一帮只设置100token 。

配置代码来说 token都特别长，订单服务4个字还是无法覆盖代码的后半段。

这种情况叫做**层级语义丢失**。

1. 结构化细分与拼接：

在切分文档的时候 不能只看个数 还要识别文档的层级 比如一级标题 二级标题。在存入chunk的时候 在code前强行拼接上父级标题。彻底保留上下文。

理解为在一个chunk前带上了家谱。精确召回相关的文本，彻底保留上下文。

2. 元数据挂载 

纯向量检索 不够精确 相当于是大海捞针 万一捞出哦了。

向量数据库在底层使用的是数学上的空间距离 也就是相似度。

比如用户： 帮我查一下 a的b 

纯靠相似度检索 可能会吧c的b d的b都找出来 

所以 应该使用 **向量+传统结构化查询**

向量数据库不只存 向量。 完整数据结构：{ ID, Vector (向量数组), Text (原文本), Metadata (JSON对象) }

在构建知识库/切分文档的时候 可以利用llm or 正则提取 给每一个chunk 打上一个tag 纯净 metadata里。

比如：

```json
{
  "id": "1002",
  "vector": [0.12, -0.45, 0.88, ...], 
  "text": "第一章... 降级策略配置：[代码行]...",
  "metadata": {
    "doc_name": "后端架构手册",
    "business_line": "订单服务",   // <-- 这个就是挂载的元数据
    "update_time": "2026-03-01",
    "author": "架构组"
  }
}
```

检索的时候，先不去查向量 根据代码逻辑 识别出用户意图（提取关键字 or 用小模型做意图分类 ） 发现用户制定了业务线 先去metadata标签里找出合适的 使用标量过滤 再做向量检索TOPK。
