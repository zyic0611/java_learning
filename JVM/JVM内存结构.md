# JVM内存结构

## 组成

1. 堆：最大的一片内存区域，被所有线程共享，氛围老年代和新生代。新生代氛围Eden区和两个Survivor区
2. java虚拟机栈：每个线程都会有自己的独立java虚拟机栈 生命周期与线程一样。每个方法执行都会创建一个栈帧，存储：局部变量表，操作数栈，动态链接，方法出口等信息。
3. 本地方法栈：jvm会使用到的native方法服务。本地方法执行也会创建栈帧，hotspot vm 中 与java虚拟机合二为一。
4. 方法区/元空间 ：逻辑上属于jvm，实际上放在本地内存里。用于存储vm加载的类信息，常量，静态变量等数据。有堆的逻辑部分的别名，非堆的别名。可以选择不实现GC。
5. 程序计数器：每个线程私有，可视为当前线程所执行的字节码文件行号指示。存储当前执行的jvm指令地址。调用native方法，计数器为null。



## 元空间 方法区 永久代

**概念与实现的关系**：

- **方法区（Method Area）**：是 JVM 规范中的一个**逻辑概念**，负责存储类信息、常量、静态变量等。
- **永久代（PermGen） / 元空间（MetaSpace）**：是 HotSpot 虚拟机对方法区的**具体物理落地实现**。

**JDK 的历史演变（核心考点）**：

- **JDK 7 及以前**：方法区的实现叫“永久代”，物理上位于 JVM 的堆内存中。**缺点**：极其容易发生 `OutOfMemoryError`，因为分配给它的空间很难动态调整。
- **JDK 8 及以后**：彻底废弃了永久代，改用“元空间”来实现方法区。**核心变化**：元空间不再使用 JVM 分配的内存，而是直接使用**本地内存（Native Memory）**。只要你电脑物理内存足够大，元空间就很难再发生 OOM。

## 方法区内容

1. 类结构，父类，实现接口 修饰符(造对象的蓝图)
2. 静态变量
3. 执行代码  方法字节码指令、JIT 编译后的代码缓存。
4. 运行时常量 由 Class 文件常量池加载而来，存放各种字面量和符号引用，支持运行期间动态生成常量。

**注意**:

虽然逻辑上静态变量属于方法区，但在 JDK 7 及以后的实际物理落地中，HotSpot 虚拟机已经把静态变量和字符串常量池挪到了堆里。

### 运行时常量池和常量池

**常量池**：存在于 Class 文件中，存储字面量和符号引用。

**运行时常量池**：当类被加载后，Class 文件里的那个静态常量池数据，就会被读到方法区里，生成“运行时常量池”。这就好比把纸质密码本录入了电脑系统，不仅能查，还能在运行期间动态往里面加新密码。

### 方法字节码和代码缓存

1. 方法字节码：

.class文件是字节码文件，里面都是字节码。

jvm启动后，会把class里的所有字节码梳理一遍，存入方法区，也就是方法字节码。此时不仅有字节码指令，还有索引和关联。

是给jvm解释器看的 边读边翻译给cpu

2. jit编译后的代码缓存

如果一个方法被跑了太多次，JVM 会觉得翻译字节码太慢了，直接把它翻译成 CPU 能看得懂的0101机器码,这部分也存在方法区，但不是字节码了。



3.解释和编译

解释就是把字节码解释成cpu看的懂的机器码 编译就是直接运行jit即时编译器生成的机器码缓存

所以java是半编译 半解释语言
