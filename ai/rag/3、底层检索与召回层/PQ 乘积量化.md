# PQ 乘积量化

Product Quantization 乘积量化 解决内存问题

一个1536维的向量 每个维度都是32位浮点数 也就是4Bytes 1536*4 约等于6kb

如果有一亿个向量 就是600GB 必须压缩。

PQ的核心逻辑是 切分（分治）+局部kmeans

1. 切割向量 

   - 把一个长度为1536（L）的向量 均分为 M个 子空间

   - 比如分为96段 每段就是16维度

2. 局部建库

   - 对于所有向量 比如1e个 的第一段 使用k-means聚成256个中心点 

   - 给这些点编号0-255  刚好可以用1B（8bit）来存储  称之为第一段的密码本 每一段都会有一个密码本

3. 量化替换（压缩）

   - 把一个原始向量拿出来 先切成96段 然后第一段就去第一段的密码本里寻找最近的点 

   - 然后把16维浮点数 转成密码本的点编号。

   - 则最后 1536维浮点数 转成96维整数  压缩率高达64。



## ADC 非对称距离计算

当查询向量的时候 不可能把所有rag里的向量 还原回1536维度。

此时采用adc   ：

1. query 切片: 先把query切成96片 和pq的处理一样

2. 然后拿出每一片 去密码本里的256个中心点（浮点数） 埃个计算距离 得到256个距离 存为数组

3. 以此类推 得到了 96段x 256个距离。 称之为 距离查表 LookUp Table LUT。二维表格。

 性能推演：96x256=24576次低维点积 在cpu里运算很快。

4.  还是需要把rag里1e个压缩后的向量 与query进行计算距离 
5. 计算距离就靠查表 做加法。 比如rag里的一号向量 分为96段 每一段都是一个int整型 代表第一段的密码本里的序号  比如是81 
6. 就直接去LUT里 拿出第一行 第81列 也就是query的第一段与 第一段的81号的距离 就行了
7. 同理 查表拿出96个距离 累加即可。 



逻辑核心：

原本计算：一个向量和query的距离需要做1536次 浮点数乘法 再做相加

ADC计算： 变成了96次数组寻址+95次浮点数加法 

后端分析： 并且对于后端来说 乘法指令消耗的cpu比加法大。 而且LUT表很小 可以放进cpu的l1 cache 则96次读内存的操作 会非常快。
