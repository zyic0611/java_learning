# 概念与架构

rag retrieval-augmented generation 检索增强生成 

本质上是给llm 开卷考试 

llm的知识只停留在训练结束的一天 并且不知道一些特定的私密数据。

## rag解决三大问题

1. 幻觉 ：强迫模型基于检索出的客观事实回答
2. 数据时效性
3. 数据安全与隐私 私密数据不能给llm公开训练。 rag相当于建来一个私有资料库给模型查阅。   并且rag可以做到 鉴权检索 不同的用户只能检索生成对应的数据。



## 链路架构

rag系统严格氛围 离线【数据入库 && 在线【检索生成

### 离线数据流 建立索引

1. 数据提取：把各种格式的文档 pdf word 网页 mysql数据 清洗提纯成纯文本
2. 分块  把长文本 分成一段一段的 chunk。 因为llm有上下文长度限制 【token限制  并且所有文本来计算向量不准确 切分为段落比较准确。
3. 向量化 调用embbedding 模型 把文本段落 变成向量 也就是高纬度浮点数数组
4. 入库 把原文本和对应向量数据存入向量数据库  比如 Milvus Qdrant or 带vector插件的redis postgresql

### 在线业务流 查询与生成

1. 用户提问 输入问题
2. 查询向量化 使用和离线阶段相同的 embbeding model 问题转变为向量
3. 向量检索 在vector db 中计算所有向量与问题向量相似度【基于KNN or 余弦相似度 召回最相似的Top-K文档块
4. prompt组装（augmentation 把找回的topk文本作为参考背景 和用户原问题拼接成一个巨大的prompt



## 与微调对比

微调（Fine-tunning） 

理解为微调是去 专项训练 rag是给一本带目录的参考书

rag：

- 知识更新频率很高 随时增删改数据集 实时生效
- 擅长外部知识 可以直接引用外部文档 可溯源。特点是外部事实的精准召回。
- 定制语气/风格比较弱  主要根据prompt引导
- 算力与成本低 只要存vector和调api
- 回答可溯源 明确告诉用户回答来自哪个文档那一页

fine-tuning

- 更新频率低 每次更新需要重新准备数据集和重新训练
- 外部知识效果很差 模型容易忘记具体的数值和细节 容易产生幻觉
- 定制风格语气很强 很适合让模型学习特定的黑话 文风 或者特定的输出格式
- 成本很高 需要准备高质量的QA数据集 消耗大量GPU算力
- 回答不可溯源 模型把知识融入权重参数里 找不到出处



二者不是互斥，学知识 查询数据用rag 定风格 学垂直领域用finetunning 。高阶工程一般用 微调基座模型+rag挂载外部知识库组合使用。
