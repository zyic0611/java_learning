# study-list

### 第一部分：RAG (检索增强生成) 深度逻辑拆解

单纯的“向量化 -> 存库 -> 相似度检索 -> 丢给大模型”只是玩具级别的 RAG。工业界和面试中看重的是各个环节的优化和信息论视角的理解。

**1. 召回前的处理：数据流转与语义边界**

- **文档解析 (Parsing) 的信息丢失率：** PDF/复杂网页解析时，如何保证表格 (Table) 和图文混合内容的结构语义不被破坏？
- **分块策略 (Chunking) 的逻辑推演：** 为什么按字数截断是最差的方案？深入理解基于特定规则（如标点、段落）和基于语义模型（计算相邻句子的嵌入余弦相似度，遇低谷则切分）的分块逻辑。
- **Embedding 模型的本质：** 了解 Word2Vec 到 BERT 再到 Contrastive Learning (对比学习) 的演进。理解为什么 Embedding 向量能够在高维空间中表征语义距离。

**2. 检索底座：向量数据库的核心算法**

- **ANN (近似最近邻) 搜索：** 为什么工业界不用精确的暴力穷举 (Flat 索引)？
- **HNSW (分层可导航小世界) 算法推演：** 从跳表 (Skip List) 的一维逻辑推演到多维图结构。理解 HNSW 中节点插入、连边规则以及不同层的查找路径。
- **IVF-PQ (倒排索引+乘积量化)：** 如何通过聚类 (K-means) 减少搜索范围，以及如何通过乘积量化压缩高维向量的内存占用。

**3. 检索进阶：如何打破 Query 的语义局限**

- **Query 重写与扩展：** 用户输入往往很短且口语化。理解 Multi-Query（大模型改写多个相似 Query 分别检索再取并集）和 HyDE（假设性文档嵌入：先让大模型生成一个虚构答案，用答案的向量去检索）的运作机制。
- **混合检索 (Hybrid Search)：** 为什么 Dense Retrieval（稠密/向量检索）会漏掉特定的专有名词或编号？梳理 Sparse Retrieval（稀疏检索，如 BM25 算法的 TF-IDF 变体计算逻辑）与向量检索的互补关系。

**4. 召回后处理：信息噪音的过滤**

- **重排机制 (Reranking)：** 为什么向量检索出来的 Top-K 顺序不一定最符合人类意图？理解 Bi-Encoder（双塔模型，快但粗糙）与 Cross-Encoder（交叉编码器，慢但精准，适合做 Rerank）的架构差异。
- **上下文窗口管理：** 探讨大模型的“Lost in the middle”（迷失在中间）现象。如何通过动态截断、上下文压缩来优化最终送给 LLM 的 Prompt。

**5. 前沿 RAG 形态**

- **GraphRAG：** 知识图谱 (实体关系抽取) 与向量检索的结合，如何解决跨文档的全局多跳推理问题。
- **Self-RAG：** 引入大模型的自我反思机制（生成 Retrieval 标记判断是否需要检索，生成 Critique 标记评估检索内容的相关性和生成质量）。



### 第一层：数据接入与索引工程层 (Data Ingestion & Indexing)

这一层解决的是：**“如何把复杂的非结构化人类知识，变成机器能看懂的高质量数据结构？”**

- **✅ 你已经掌握的：**
  - **Chunk 的策略：** 知道不能死板切分，理解了基于语义或规则的切分逻辑。
  - **结构化细分与拼接：** 如何保持文档上下文的连贯性。
  - **元数据挂载 (Metadata)：** 给 Chunk 打上时间、作者、分类等 Tag，这是后续做高精度过滤的基石。
- **❌ 你需要补足的盲区：**
  - **复杂版面解析 (Layout Analysis / Vision-parsing)：** 面对带有多级表头、跨页表格、图文混排的 PDF，单纯的文本截取会破坏二维结构。你需要了解如何用视觉模型（如 LayoutLM）或 OCR 技术把表格转为 Markdown 或 HTML 格式再进行切分。
  - **GraphRAG (知识图谱索引)：** 传统的 Vector Chunk 解决不了跨文档的“全局宏观问题”。你需要了解如何用大模型从文本中抽取“实体”和“关系”，构建图索引。

### 第二层：查询理解与流量分发层 (Query Translation & Routing)

这一层解决的是：**“用户的问题往往很简短或模糊，大模型怎么知道他真正想要什么？”**

- **✅ 你已经掌握的：**
  - **意图提取：** 分析用户 Query 的真实目的。
  - **Prompt 查询重写 (Query Rewrite)：**
    - *HyDE (假设性文档嵌入)：* 让大模型先“盲猜”一个答案，用答案的向量去检索。
    - *Multi-Query (多路查询)：* 让大模型用不同的表述把一个问题改写成多个，分别检索再取并集。
- **❌ 你需要补足的盲区：**
  - **查询路由 (Query Routing)：** 用户问“帮我总结一下全书大意”和问“张三在第三章说了什么”，系统不应该走同一条检索链路。你需要了解逻辑路由算法，决定流量是去查向量库、查 SQL 数据库（Text-to-SQL）、还是去查知识图谱。
  - **子查询拆解 (Sub-Query / Least-to-Most)：** 针对多跳问题（如“A公司去年营收和B公司前年营收差多少”），如何让模型拆解为两个独立的查询去执行。

### 第三层：底层检索与召回层 (Retrieval Engine)

这一层解决的是：**“如何在毫秒级时间内，从海量数据中把高度相关的知识捞出来？”**

- **✅ 你已经掌握的：**
  - **向量库核心算法：** 掌握了 ANN、HNSW (分层图网络)。
  - **性能与存储压榨：** 掌握了 PQ 乘积量化以及 ADC (非对称距离计算) 的底层查表逻辑。
- **❌ 你需要补足的盲区：**
  - **混合检索 (Hybrid Search) 与 RRF 算法：** 向量检索（Dense）对专有名词、特定编号（如型号、UUID）极度不敏感。工业界绝对不能只用向量检索，必须结合传统的**稀疏检索 (Sparse Retrieval，如基于 TF-IDF 的 BM25 算法)**。同时，你需要了解 **RRF (倒数秩融合)** 算法，它是如何把 BM25 搜出来的 Top-K 和向量搜出来的 Top-K 科学地合并在一起的。

### 第四层：召回后处理层 (Post-Retrieval)

这一层解决的是：**“捞出来的数据往往有噪音且太长，怎么在喂给大模型前进行精炼？”**

- **✅ 你已经掌握的：**
  - **动态过滤：** 结合元数据 (Metadata) 进行硬性条件筛选。
  - **双重重排 (Double Reranking)：** 使用 Cross-Encoder (交叉编码器) 模型对初筛的 Top-K 重新打分排序。
  - **上下文压缩 (Context Compression)：** 剔除掉 Chunk 中对回答问题无用的冗余句子，节省大模型 Token 并缓解“迷失在中间”的注意力涣散现象。
- **💡 点评：** 这一层你的笔记非常完整，覆盖了目前业界最核心的提纯手段。

### 第五层：生成与自适应闭环层 (Generation & Control)

这一层解决的是：**“拿到精炼后的数据，大模型怎么组合答案？如果检索回来的数据全是错的怎么办？”**

- **✅ 你已经掌握的：**
  - **LLM 节点：** 如何把检索到的 Context 和用户的 Query 拼接进 Prompt，约束大模型进行事实性输出。
- **❌ 你需要补足的盲区：**
  - **自适应 RAG (CRAG / Self-RAG)：** 这是进阶的分水岭。标准的 RAG 是“检索啥就用啥回答”。但如果检索系统今天“翻车”了，没捞到相关内容呢？你需要了解如何训练或提示大模型去**评估检索质量**（生成 Retrieval 标签），如果发现全是无用信息，大模型要能够主动发起二次网络搜索，或者直接回答“基于当前文档无法回答”，而不是强行胡编乱造。
  - **RAG 评估指标 (RAGOps)：** 作为工程师，你怎么知道你的系统好不好？不能凭感觉。你需要了解 **RAGAS** 或 TruLens 等评估框架中的核心量化指标，比如：*Context Precision (上下文精准度)*、*Context Recall (上下文召回率)*、*Faithfulness (忠实度/幻觉率)*。
