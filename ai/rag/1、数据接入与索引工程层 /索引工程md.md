# 索引工程

## 问题

在处理那本 500 页《后端架构手册》时，遇到了一个 Bug。

> 手册第 30 页有一个大标题叫 **【订单服务的降级策略】**，下面跟着一段具体的配置代码和重试机制的描述。
>
> 业务同学提问：‘订单服务怎么配置重试机制？’ 结果系统没找对答案。我们排查后发现，切割的时候，**【订单服务的降级策略】** 这个标题被分到了 Chunk 1，而 **具体的配置代码和重试机制** 被分到了 Chunk 2。因为用户的提问里有‘订单服务’，所以系统去查的时候，Chunk 2 里只有干巴巴的代码，没有‘订单服务’这四个字，导致相似度极低，没有被召回。
>
> **请问，如果遇到这种‘标题和正文被跨 Chunk 切断，导致正文丢失了上下文（不知道这属于哪个业务线）’的情况，你有什么在工程上容易落地的解决方案吗？**”



## 解决

首先**overlap**是必须的 但是overlap一般只设置100token 。

配置代码来说 token都特别长，订单服务4个字还是无法覆盖代码的后半段。

这种情况叫做**层级语义丢失**。

### 结构化细分与拼接：

在切分文档的时候 不能只看个数 还要识别文档的层级 比如一级标题 二级标题。在存入chunk的时候 在code前强行拼接上父级标题。彻底保留上下文。

理解为在一个chunk前带上了家谱。精确召回相关的文本，彻底保留上下文。

### 元数据挂载 

纯向量检索 不够精确 相当于是大海捞针 万一捞出错了。

向量数据库在底层使用的是数学上的空间距离 也就是相似度。

比如用户： 帮我查一下 a的b 

纯靠相似度检索 可能会吧c的b d的b都找出来 

所以 应该使用 **向量+传统结构化查询**

向量数据库不只存 向量。 完整数据结构：{ ID, Vector (向量数组), Text (原文本), Metadata (JSON对象) }

在构建知识库/切分文档的时候 可以利用llm or 正则提取 给每一个chunk 打上一个tag 纯净 metadata里。

比如：

```json
{
  "id": "1002",
  "vector": [0.12, -0.45, 0.88, ...], 
  "text": "第一章... 降级策略配置：[代码行]...",
  "metadata": {
    "doc_name": "后端架构手册",
    "business_line": "订单服务",   // <-- 这个就是挂载的元数据
    "update_time": "2026-03-01",
    "author": "架构组"
  }
}
```

检索的时候，先不去查向量 根据代码逻辑 **识别出用户意图（提取关键字 or 用小模型做意图分类 ）** 发现用户制定了业务线 

先去metadata标签里找出合适的 使用标量过滤   称为预过滤

再做向量检索TOPK。
