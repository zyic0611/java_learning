# 参数提取与幻觉控制

也就是容错与兜底机制 。

大模型不一定每次输出都是标准的json schema去调用对应tool 则需要防御性编程。

## 参数缺失处理

当userinput模糊输入的时候 模型决定调用某个tool 但是发现某个必须参数缺失 

错误： 后端直接跑出错误 或者1返回500

使用状态机+反向prompt ：后端解析到toolcall请求 校验发现参数缺失 则不执行tool 构造一条system message伪装成工具调用的结果给llm（正常交互则直接返回工具调用结果让llm润色给用户） lllm接收到这个信息 则生成一个话回复用户 提示用户补足参数 在多轮对话中实现参数补齐。



### 格式幻觉修复

就算用json schema约束 较弱的模型在生成tool call的请求给后端调用接口的时候 仍然可能输出不干净的

 比如（带有 Markdown 标记的 JSON（如 `json { ... } `），甚至在 JSON 结尾加上“这就是你要的参数”）

- **第一道防线（正则与字符串清洗）：** 使用正则表达式，精准剥离首尾的 Markdown 标记（`json 和 `），并忽略 JSON 结构之外的任何自然语言废话。

- **第二道防线（宽容的解析器）：** 在后端反序列化时，配置宽松模式。比如在使用 Jackson 库时，开启允许单引号、允许未引用的字段名、忽略未知属性等特性（`ALLOW_SINGLE_QUOTES`, `FAIL_ON_UNKNOWN_PROPERTIES=false`），尽最大可能解析残缺数据。

- **第三道防线（AST 修复与 LLM 纠错）：** 如果 JSON 彻底损坏（比如少了括号），可以使用基于 AST（抽象语法树）的 JSON 修复算法补全括号。如果依然失败，直接捕获异常，将异常信息（如 `JSON parse error at line 3`）发还给大模型，让大模型自己重新生成一次（Retry 机制）。



## 隔离与沙盒防御

当 Agent 拥有了编写并执行代码的能力（Code Interpreter），风险就呈指数级上升。如果不做隔离，模型生成的代码可以直接读取后端环境变量（如数据库密码），甚至执行破坏性命令。

###  Code Interpreter (代码解释器) 的必要性

大模型其实不擅长做复杂的数学计算、数据分析或精准的数据过滤。遇到这些问题，最好的方案是让 LLM 生成一段 Python 脚本，由后端执行这段脚本拿到精确结果，再交回给 LLM。

### Docker 沙盒隔离机制 (Sandbox Isolation)

这是保障系统安全的底线。你需要为代码执行构建一个“插翅难逃”的环境。

1. **进程级隔离（Docker / gVisor）：** 千万不要在后端宿主机直接 `Runtime.getRuntime().exec("python script.py")`。每一次代码执行，都应该跑在一个临时拉起的轻量级 Docker 容器中。对于更高安全级别的 SaaS 平台，甚至需要用 gVisor 这种用户态内核进行更底层的系统调用拦截。
2. **网络隔离 (Network Egress Control)：** 沙盒默认**断网**。如果代码只是做计算和数据处理，不需要外网。如果必须要下载文件，只允许访问白名单内的域名，防止代码将机密数据 POST 到黑客的服务器。
3. **权限剥夺 (Non-Root User)：** 容器内的执行用户绝对不能是 root。创建一个低权限用户（如 `nobody` 或特定的 `runner` 用户），剥夺读写系统关键目录的权限。
4. **资源配额限制 (cgroups)：** 防止模型写出死循环（如 `while(true)`）或内存泄漏代码导致服务器雪崩。通过 Docker 的 cgroups 限制 CPU 使用率（例如最多 0.5 核）和内存（例如最大限制 128MB）。
5. **超时机制 (Timeout Kill)：** 任何代码执行任务，必须设定硬超时（如 5 秒）。时间一到，无论代码是否跑完，后端强制向容器发送 SIGKILL 信号销毁进程。可以将这类高耗时的代码执行任务扔给消息队列（例如 RabbitMQ）去异步处理，避免阻塞主业务线程。
6. **只读文件系统 (Read-only Filesystem)：** 将容器的根文件系统设置为只读 (`--read-only`)，仅挂载一个极小的临时挂载点 (`tmpfs` 或特定 volume) 供脚本输出结果文件。代码跑完，容器和临时挂载点一起销毁，做到“阅后即焚”。
