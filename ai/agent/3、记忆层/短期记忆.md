# 短期记忆

短期记忆就是 直接拼接再prompt里 喂给大模型的上下文 。    

由于大模型有上下文窗口的限制 并且token计费。

该层核心 解决 有限的空间 保留最高质量的信息。

也 就是在记忆够多和不撑爆上下文之间取舍 是记忆工程的核心

一般存在redis或者应用内存里 java的`list<message>`





## 滑动窗口 与 token截断

维护一个固定大小队列 一般大小按照token衡量。 并不是轮数 因为每轮对话长短不一。 

重点：

- System Prompt丢失 ：  不能粗暴的对整个context列表进行截断 通常首尾保留 固化System Prompt 有时还包括最初的几轮关键设定 中间的历史对话采用滑动窗口
- Lost in the Middle ： 中间注意力丢失 大模型对首尾消息敏感 对中间消息迟钝 如果关键指令再滑动窗口中间 可能会被llm 忽略
- 不对称截断： 一轮完整的对话包括 user & assistant的交互 截断的时候必须保证 QA的完整性 不能缺失其一。



## 动态摘要压缩

当滑动窗口快满的的时候 触发一个后台任务 调用小模型or当前模型 将早期的N轮对话 压缩成summary 然后清空N轮 使用summary代替

重点：

- 信息降维与细节丢失： summary本质是有损压缩  前面对话一些比较具体的 代码段 订单号 细微的情感变化 压缩后 会丢失细节
- 多级摘要： 高级的agent 会做 层级摘要。 比如10条对话->一条小结 5个小结->一个阶段性总结 。采用树状合并 拉长上下文跨度
- 异步调度： 后端设计中 摘要计算非常耗时，不能在用户等待回复的链路中做摘要。 通常丢给MQ进行异步处理
