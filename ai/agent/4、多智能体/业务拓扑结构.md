# 拓扑结构

不同结构的设计模式 底层的执行引擎/控制流框架 都是图状态机。

## 串行SOP
也称为pipeline 流水线模式 

类似 MetaGPT，明确产品经理、架构师、程序员角色，**上游的输出严格作为下游的输入。**

如果是纯黑盒llm 是靠prompt告诉模型 一些串行操作 实现的 容易出错。

在图状态机中 退化为DAG。 节点执行完 由普通边传递状态给下一个节点  执行过程由图引擎硬编码。

举例：

**节点 (Node)：** `PM_Node`, `Architect_Node`, `Coder_Node`。

**边 (Edge)：** `PM_Node` 执行完毕后，一条**普通边 (Normal Edge)** 强制将 State 传递给 `Architect_Node`。没有任何 LLM 可以决定下一步去哪，完全由图引擎硬编码锁死。

## 层级管理
也称为主管/路由模式 supervisor

优点： 能处理复杂的模糊任务 扩展性强 加一个性能力 加一个worker

缺点：supervisor的能力决定系统上线 他的路由错了 就都错了

图引擎中的 树状结构/星状结构。

一个Manager负责拆解任务 分发给多个worker 最后汇总验收。

在图状态机中 一般是带有条件路由的复杂图。

举例：

**节点：** `Manager_Node`, `Worker_Frontend`, `Worker_Backend`。

**流转逻辑：**

1. `Manager_Node` 拆解完任务写入 State。
2. 图引擎触发**条件边 (Conditional Edge)** 检查 State：如果任务标签是“写接口”，就路由到 `Worker_Backend`；如果是“画页面”，就路由到 `Worker_Frontend`。
3. Worker 执行完后，边将状态流转**回** `Manager_Node` 进行验收。这就形成了图里面的**回路**。



## 辩论与博弈

图引擎中带有 跳出条件的 环状图。

架构逻辑：两个agent相互对抗or协作。生成器generator 批评者reviewer 。生成的东西给reviewer去测试 如果错了 就拿错误日志给generator重新生成 直到reviewer满意。

优点：降低大模型幻觉最有效

缺点：token消耗巨大 没有优秀的停止条件 容易死循环



举例：

**节点：** `Agent_A` (提方案), `Agent_B` (找茬)。

**流转逻辑：**

1. `Agent_A` 输出方案 -> `Agent_B` 提出质疑。
2. 触发**条件边**检查 State 中的 `consensus_reached` 字段（是否达成共识）：
   - **如果 False：** 带着质疑意见，沿着边**倒流回** `Agent_A` 让它修改（这就是状态机里的一个循环 Loop）。
   - **如果 True：** 跳出循环，走向 `End_Node`。
3. **熔断机制：** 图引擎会在边上加一个计数器 `retry_count`，如果循环超过 5 次还没共识，图引擎直接掐断，强制抛出异常（防止把 API 余额耗尽）。



## 提升

Q： 多个agent之间是共享一个 context or 各自独立

A： 按需分配。 pipeline 通常传递summary节约token 不是传递全亮对话。  supervisor 模式 主管拥有全局context 具体woker只拥有需要的局部信息

Q：怎么防止multi agent进入 死循环

A： 工程上要有兜底机制

1. 硬性截断：状态机维护最大循环次数 超过5n次抛异常or 转人工
2. 强制仲裁者 两个agent僵持不下 引入第三方配置更高算力模型 的裁判agent进行最终裁决
